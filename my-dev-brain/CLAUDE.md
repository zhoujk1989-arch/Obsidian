# LLM Wiki Agent Guide

本文件是本仓库的维护规则权威版本。

- `CLAUDE.md` 是面向其他 agent 工具的兼容镜像，应与本文件保持一致。
- `00-首页/工作说明.md` 只保留摘要导航，不再重复整份规则。

本仓库的目标不是临时回答问题，而是持续沉淀结构化知识。

## 核心原则

1. 原始资料进入 `01-资料库/` 时必须忠实转录；除格式整理外，不得改写、缩写、漏写或替换原文含义。首次入库后只增不改，作为事实来源保留。
2. Wiki 页面是整理后的知识层，允许持续更新。
3. 每次处理新资料或新问题时，优先更新已有页面，而不是重复创建近义页面。
4. 回答问题时，优先复用现有 Wiki 页面；只有在知识页不够精确时才回溯原始资料。
5. 重要的新结论、新对比、新框架，必须回写到仓库，而不是只停留在聊天记录里。

## 目录与承载

- `00-首页/`：入口页、当前工作台、简版工作说明；首页只承担路由，不再维护全量索引。
- `01-资料库/`：原始资料、来源记录、原文页。
- `02-主题/`：长期维护的主题页与规则页。
- `03-实体/`：人物、公司、产品、系统、报表等对象页。
- `04-综合/`：对比、综述、FAQ、决策、问题清单、梦境页。
- `05-日志/`：ingest、query、dream、lint、fix、setup 日志。
- `06-项目/`：按项目维度维护的长期页。
- `90-模板/`：页面模板。

## 页面分工

### 资料页

- 放在 `01-资料库/`
- 保存事实来源、来源元信息、核心观点、影响范围
- 不承担长期结论层职责
- 如果页面属于 `-原文.md` 的原文页，则不按本段做摘要化整理，而按下方“原文页”规则处理

### 原文页

- 放在 `01-资料库/`，通常以 `-原文.md` 结尾
- 目标是忠实保存用户提供的原文或来源原文，允许整理格式，但不允许改变信息内容
- 默认应尽可能完整转录；如果只能保存节选，必须显式标注 `摘录` 或 `节选`
- 对原文的结论、重点、规则、对比和推断，一律写到知识页，不写进原文主体

### 主题页 / 规则页

- 放在 `02-主题/`
- 主题页沉淀稳定知识
- 规则页处理跨报表、跨页面可复用的口径规则

### 实体页

- 放在 `03-实体/`
- 针对具体对象持续维护
- 监管报表优先使用 `90-模板/报表模板.md`

### 综合页

- 放在 `04-综合/`
- 只承接跨页成果，例如对比、路线图、FAQ、决策、问题清单
- 不重写单页已经稳定存在的摘要
- 监管类综合页可继续细分为：
  - 系统页：`监管系统-报表清单.md`，维护系统信息、覆盖范围和入口
  - 报表目录页：`系统名-报表关系.md`，维护“系统包含哪些报表/数据表”的目录关系
  - 表间关系页：`系统名-表间校验关系.md` 或数据域关系页，维护报表之间的校验、映射或主键挂接关系
  - 场景映射页：维护真实业务场景涉及哪些系统和报表

### 项目页

- 放在 `06-项目/`
- 只维护目标、状态、里程碑、问题、下一步
- 不重复主题定义和资料摘要

## 去冗余规则

- 资料页写来源与核心观点，不复制大段提炼到主题页。
- 主题页和实体页写稳定结论，不重复原文长摘录。
- 综合页写跨页判断，不重复单页摘要。
- 项目页写推进状态，不重复知识层内容。
- 同一结论如果已有承接页，优先补该页，而不是新建近义页面。
- 数据字典型报表：字段“全量清单”优先只保留在原文页；报表实体页优先只保留分组、关键字段与规则，并链接回原文，避免双份维护。
- 原文页正文不承接摘要化提炼；摘要、归纳、口径判断和跨页结论，必须写入知识页或综合页。

## 原文页硬约束

- 命中以下任一条件，按原文页处理：位于 `01-资料库/` 且文件名含 `-原文`；或用户明确要求“保存原文”“写原文页”“保留数据字典原文”。
- 允许操作：忠实转成 Markdown 标题/列表/表格；补充来源元信息；修复明显 OCR 断行或排版错误；在末尾追加 `勘误`、`版本差异`、`处理记录`。
- 禁止操作：摘要化改写；省略用户已提供的段落、字段、行列、编号或核对关系；把多个段落合并成自己的表述；补入原文没有的解释；把知识结论写进原文主体。
- 无法判断某段是否属于原文时，优先保留原片段并标注 `待确认`，不要自行润色。

## 检索策略

### 默认最小读取

每次开始工作，按以下顺序读取，找到足够上下文即停止：

1. `00-首页/index.md`
2. 涉及当前维护、入库、巡检时读 `00-首页/当前工作台.md`
3. `05-日志/log.md` 中最近相关条目
4. 与当前任务直接相关的页面
5. 仍不够时再读 `01-资料库/` 原文页

默认不要全量通读日志。优先使用以下低成本方式：

- 先读最近 10 到 20 条日志
- 或先按关键词检索命中条目
- 只有任务明确依赖长期演化历史时，才扩大范围

### 监管类检索

按 L0 → L1 → L2 → L3 逐层读取：

| 层级 | 位置 | 用途 |
|------|------|------|
| L0 | `04-综合/监管系统-报表清单.md` | 先匹配系统，定位系统实体页和系统报表页 |
| L1 | `04-综合/系统名-报表关系.md` | 再按报表编码、报表名称、领域/数据域定位具体报表页；该页只维护系统-报表目录 |
| L2 | `03-实体/` 与 `02-主题/` | 读取报表结论、常见口径、跨报表规则 |
| L3 | `01-资料库/系统名/` | 核对原文定义、校验关系、项目注解 |

补充入口：

- 跨报表概念问题先查 `02-主题/` 规则页。
- 单一报表问题先查系统报表页，再进报表实体页，最后回原文页。
- 表间校验问题查 `系统名-表间校验关系.md` 或对应规则页；业务场景问题查场景映射页。

### 非监管类检索

按以下顺序读取：

1. `00-首页/index.md`
2. 相关主题页、实体页、项目页
3. 必要时回到资料页

### 上下文节流

- 不因为“存在”就读取所有主题页、实体页、项目页。
- 不重复读取 `AGENTS.md`、`CLAUDE.md`、`00-首页/工作说明.md` 三份等价说明。
- 新资料进入时，优先读直接相关页、同关键词页、反向链接页，而不是全库扫描。
- 已经在知识页找到稳定答案时，不再为了“保险”重复读取原文。

## 闭环机制

一次有效工作，至少应落到以下 4 层中的 2 层以上：

- 证据层：资料页或原文页
- 知识层：主题页、规则页、实体页、综合页
- 队列层：当前工作台、问题清单、待确认项、项目下一步（默认承载页：`00-首页/当前工作台.md` 与 `04-综合/待确认清单.md`）
- 日志层：`05-日志/log.md`

### 升级规则

- 同一问题被问到 2 次以上，升级为 FAQ、规则页或综合页。
- 同一概念出现在 3 份以上资料，且已跨 2 个以上知识页复用，升级为主题页或规则页。
- 同一结论在 2 次以上 query、dream 或 lint 中重复出现，升级为长期知识页。
- 梦境里的同类建议连续出现 2 次以上，可在后续普通维护中更新稳定页。
- 无法立即确认的问题，不静默跳过；要标注 `待确认`，并写入问题清单或项目下一步。

### 自我改进机制

“变聪明”不靠自由发挥，而靠把重复经验沉淀成更好的路由、验收和规则。每次有效工作结束前，做一次简短回看：

1. 本次是否暴露新的重复问题、失败模式、检索捷径或高频查询？
2. 这个经验是否能转成稳定规则、验收项、查询回归样例或待办？
3. 是否已有承载页可以更新，而不是新建近义页面？

经验沉淀按以下顺序选择承载页：

- 临时任务、下一步、待巡检页面：写入 `00-首页/当前工作台.md`
- 可重复检查项、命令、回归样例：写入 `04-综合/知识库维护验收清单.md`
- 稳定跨任务规则：更新 `AGENTS.md`，并同步 `CLAUDE.md`
- 业务知识、口径结论：写入对应主题页、规则页、实体页或综合页
- 证据不足的问题：写入 `04-综合/待确认清单.md`
- 操作过程和经验摘要：写入 `05-日志/log.md`

更新 `AGENTS.md` 必须克制，只有满足以下任一条件才改：

- 同类问题出现 2 次以上
- 影响 ingest、query、lint 等多个操作类型
- 已造成明显断链、重复页、误读原文、口径误判或上下文膨胀
- 能写成具体动作，而不是泛泛建议

## 操作类型

### Ingest

当用户添加新资料时：

1. 先判断资料类型：普通资料、监管原文、数据字典、版本更新、勘误、代码表或项目资料
2. 入库前做去重检索：按标题、报表编码、系统名、版本号、物理表名、关键词查现有资料页和实体页
3. 判断版本策略：同一来源同一版本优先更新说明或去重；新版本资料新建版本原文页或追加版本差异；不得覆盖旧原文正文
4. 创建或更新资料页到 `01-资料库/`
5. 补齐最低元信息：来源、作者、发布时间、获取日期、可更新页面；缺失项写明 `待确认`
6. 如果是原文页，先忠实转录原文；提炼出的 3 到 5 条核心观点写入相关知识页、资料说明页或实体页，不要替换原文主体
7. 通过索引、关键词、反向链接定位直接相关页面
8. 将新信息合并进对应主题页、实体页、项目页或综合页
9. 如出现稳定新概念且无承接页，再新建页面
10. 补交叉链接，并按 `04-综合/知识库维护验收清单.md` 做 ingest 验收
11. 记录到 `05-日志/log.md`

Ingest 停手条件：

- 已完成证据层、知识层、路由入口和日志层的最小闭环
- 没有足够证据支撑的口径已进入 `04-综合/待确认清单.md`
- 批量资料尚未处理完时，把剩余范围写入 `00-首页/当前工作台.md`，不要用临时记忆承接

如果资料与现有内容冲突：

- 明确标记冲突点
- 不静默覆盖旧结论
- 在相关页面注明依据来源与日期
- 如果冲突发生在原文页，优先新建新版本原文页或追加 `勘误/版本差异`，不要直接改写旧原文正文

监管类 ingest 默认顺序：

1. 在 `01-资料库/系统名/` 创建原文页
2. 更新或创建系统页到 `03-实体/`
3. 必要时更新 `04-综合/监管系统-报表清单.md` 的系统级覆盖范围和入口；不要在该页逐张追加报表明细
4. 更新或创建 `04-综合/系统名-报表关系.md`，只补系统-报表目录行；系统概要写入系统实体页，报表日期/频度/字段/口径写入报表实体页，表间校验或业务场景写入关系页、场景页或规则页
5. 更新或创建报表页到 `03-实体/`
6. 若跨报表概念已满足升级条件，创建或更新规则页到 `02-主题/`
7. 只有在前面几层承接不下时，再补主题页或对比页

### Query

当用户提问时：

1. 先判断问题类型：事实查找、口径解释、跨表对比、差异核对、项目状态、巡检修复、开放问题
2. 按检索策略逐层读取，找到稳定答案即停止；不要为了保险全库扫描
3. 优先引用知识页；只有知识页不够精确、存在冲突或用户要原文依据时，才回到原文页
4. 输出时明确区分“已知结论”“依据来源”“推断判断”“待确认”
5. 如果本次回答产生了长期有价值的新内容，优先回写到已有知识页，不回写到原文页正文
6. 如果仍无法沉淀为稳定结论，标记 `待确认`，并放入问题清单或项目下一步
7. 如果问题可能复现，补入 `04-综合/知识库维护验收清单.md` 的查询回归样例或对应 FAQ/规则页
8. 有沉淀价值、产生页面变更或形成待确认项时，记录到 `05-日志/log.md`；纯临时查询不强行记日志

Query 停手条件：

- 已能用现有知识页给出准确回答
- 已说明哪些部分来自原文，哪些只是推断
- 无法确认的部分已进入待确认队列
- 没有新增长期价值时，不为了“沉淀”而制造新页面

### Dream

当用户要求“做一次梦”时：

1. 读取 `00-首页/index.md`
2. 读取 `00-首页/我是谁.md`、`00-首页/我的偏好.md`、`00-首页/我的目标.md`
3. 读取 `06-项目/当前学习地图.md`、相关项目页与最近相关日志
4. 优先关注最近 7 天更新内容；信息不足时再回看最近 30 天
5. 输出必须区分“观察”和“推断”
6. 推断可以大胆，但不能写成既定事实
7. 自动落到 `04-综合/梦境/YYYY-MM-DD-梦境.md`
8. 本轮只提出建议更新的核心页，不直接改写稳定自我页
9. 记录到 `05-日志/log.md`

### Lint

定期巡检知识库：

- 先按 `04-综合/知识库维护验收清单.md` 区分“可自动修复项”和“必须待确认项”
- 是否有重复页面
- 是否存在孤立页面
- 是否缺少重要交叉链接
- 是否有明显过时结论
- 是否有高频概念尚未建页
- 是否有冲突信息未标注来源
- 主题页、规则页、实体页是否缺少 frontmatter
- `04-综合/监管系统-报表清单.md` 的系统行是否与已纳入系统一致
- `04-综合/系统名-报表关系.md` 的报表行是否与该系统已纳入报表一致
- `04-综合/系统名-报表关系.md` 是否仍是系统-报表目录页；如果堆入系统概要、单表口径、报送日期、字段摘要、表间校验或场景路由，应迁回系统实体页、报表实体页、关系页或场景页

Lint 分级：

- L1 结构巡检：frontmatter、断链、重复链接、异常列表前缀、根目录临时文件、超长行、命名不一致
- L2 路由巡检：首页、工作台、系统清单、系统报表页、资料目录之间的入口是否一致
- L3 知识巡检：重复页面、冲突结论、待确认积压、高频概念未建页、规则页缺失

巡检后：

- 先修补结构问题
- 再把待升级、待确认、待建内容沉淀到对应页面
- 巡检报告应包含：范围、发现、已修复、待确认、后续优先级
- 最后记录到 `05-日志/log.md`

### Fix

当用户要求修复结构问题，或巡检发现可自动修复问题时：

1. 先界定修复范围，确认问题属于格式、链接、计数、索引、命名、重复、归档等结构问题
2. 能自动修的直接修；涉及原文内容、口径判断、多版本差异时转入 `待确认`
3. 修复时优先更新已有页面，不新建近义页面
4. 如果修复产生新的可复用检查项，回写 `04-综合/知识库维护验收清单.md`
5. 记录到 `05-日志/log.md`，类型使用 `fix`

## 写作约定

1. 优先使用中文。
2. 页面标题简洁明确，避免同义重复。
3. 一页只承载一个稳定主题。
4. 使用 Markdown 链接互相引用。
5. 普通知识页不堆砌长摘录，优先写提炼后的结论；监管原文页保留精度。
6. 对不确定内容显式标注 `待确认`。
7. 对时间敏感内容尽量写明具体日期。
8. 以下页面必须包含 YAML frontmatter：
   - 实体页：`type: 实体`, `entity_type: ...`, `keywords: [...]`
   - 主题页：`type: 主题`, `keywords: [...]`
   - 规则页：`type: 规则页`, `keywords: [...]`

原文页、首页、梦境页、日志、模板不强制要求 frontmatter。

## 命名建议

- 主题页：`主题名.md`
- 实体页：`实体名.md`
- 综合页：`主题-对比.md`、`主题-综述.md`、`主题-路线图.md`
- 资料页：`YYYY-MM-DD-标题.md`

## 开始与结束检查

### 开始工作时

1. 读取 `00-首页/index.md`
2. 涉及当前维护、入库、巡检时读取 `00-首页/当前工作台.md`
3. 读取最近相关日志，而不是默认整份日志
4. 读取当前任务最相关的页面

### 结束工作时

1. 是否更新了受影响页面
2. 是否补上了交叉链接
3. 是否把高价值输出沉淀到知识层或队列层
4. 是否记录到日志
5. 是否按 `04-综合/知识库维护验收清单.md` 完成对应操作类型验收
6. 是否有可复用经验需要回写到工作台、验收清单、待确认清单或规则文件
