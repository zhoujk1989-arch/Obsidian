# LLM Wiki-当前缺口与优化建议

## 背景

本页基于 [[01-资料库/2026-04-05-Karpathy-How I use LLMs|Karpathy: How I use LLMs]] 的核心原则，对当前仓库进行一次面向工作流的评估。

重点不是判断“页面多不多”，而是判断这套系统是否已经形成可持续的知识维护闭环。

## 已经做到的部分

- 已经建立原始资料层、主题层、实体层、综合层、项目层和日志层，基本符合 Karpathy 所说的“事实层 + 知识层”分离。
- 已经把重要操作写入 [[05-日志/log|日志]]，说明系统开始具备可追踪的演化历史。
- 在监管知识场景里，已经形成 [[04-综合/监管系统-报表清单|路由表]]、[[03-实体/1104系统|系统页]]、报表页和规则页，这说明知识分层不只是概念，已经在真实业务场景里落地。
- 已经有巡检页和梦境页，说明系统不只会摄入资料，也开始做结构检查和阶段性综合。

## 当前主要缺口

### 1. 缺少“活跃工作台”层

已知情况：

- [[06-项目/我的 LLM Wiki 搭建|项目页]] 和 [[README|README]] 都已经写到要建立“资料导入清单”和“每周巡检机制”。
- 但当前仓库里还没有这两类实际页面，很多待处理动作仍然散落在“下一步”“待补充”“待确认”里。

判断：

- 现在的仓库已经有“知识沉淀层”，但还缺“运营层”。
- 没有活跃工作台，系统就很依赖你和 LLM 的短期记忆，难以稳定管理“哪些资料待处理、哪些结论待升级、哪些页面待巡检”。

建议：

- 创建一个常驻工作台页，例如 `00-首页/当前工作台.md` 或 `06-项目/资料导入清单.md`。
- 最少维护 3 个区块：待摄入资料、待升级结论、待巡检页面。

### 2. 缺少“稳定结论升级机制”

已知情况：

- [[04-综合/梦境/2026-04-08-梦境|2026-04-08 梦境]] 已经明确提出，需要一层更清晰的“稳定结论升级机制”。
- 当前也已经有 query、dream、ingest 三类输出，但它们如何升级成主题页、综合页、个人页更新，还没有形成单独规则。

判断：

- 这是当前项目最核心的机制缺口。
- 如果升级规则不明确，高价值内容会停留在日志、梦境页和单次问答里，无法真正变成长期知识。

建议：

- 单独建立一页机制说明，例如 `04-综合/知识升级机制.md`。
- 明确几条升级触发条件：
  - 同一问题被问到 2 次以上，升级为 FAQ 或规则页。
  - 同一结论在 2 个以上 query 或 dream 中重复出现，升级为主题页或综合页。
  - 梦境中的建议如果连续两次出现，再进入 `我是谁`、`我的目标`、`当前学习地图` 这类稳定页。

### 3. 缺少“开放问题与待建概念”总表

已知情况：

- 当前仓库大量页面里都已经有“待补充”“待确认”“下一步”。
- 但这些开放问题还没有汇总层，只能逐页翻找。

判断：

- 这会让知识库逐渐出现隐性维护债务。
- 当系统规模继续增长时，最容易丢的不是旧资料，而是“已经意识到但还没收敛”的问题。

建议：

- 建一个 `04-综合/待确认与待建清单.md`。
- 至少按 3 类聚合：待确认结论、待建页面、高频但尚未抽象的概念。

### 4. 非监管主题的路由能力还偏弱

已知情况：

- 监管知识已经有 [[04-综合/监管系统-报表清单|L0 路由表]]。
- 但 LLM Wiki、RAG、Agent Harness、个人知识管理这些非监管主题，目前仍主要依赖 [[00-首页/index|首页]] 作为入口。

判断：

- 在当前规模下这还够用，但一旦相关资料继续增加，首页会越来越像“目录”，而不是“问题路由层”。
- 这会让查询时仍然偏向全文搜索，而不是先命中已经整理好的知识页。

建议：

- 后续可补一个轻量的非监管路由页，例如 `04-综合/LLM 与 Agent 问题索引.md`。
- 先从“常问问题 -> 推荐页面”做起，不需要一开始就做复杂检索。

### 5. 资料元信息完整度还没有变成硬约束

已知情况：

- [[01-资料库/2026-04-06-Agent Harness 讲解页|Agent Harness 讲解页]] 里仍有“作者：待补充”。
- 当前资料页结构已经很好，但元信息完整度还没有形成强制检查。

判断：

- 对于会持续回查来源的系统，元信息完整度不是装饰，而是事实层的可追溯性保证。
- 如果来源、作者、发布时间经常缺失，后续做冲突判断和时间敏感更新会更难。

建议：

- 把“来源、作者、发布时间、获取日期、可更新页面”加入 ingest 最低检查清单。
- 若暂时缺失，也应明确写“缺失原因”和“后续补充路径”。

### 6. 项目页还缺少验证指标

已知情况：

- [[06-项目/我的 LLM Wiki 搭建|我的 LLM Wiki 搭建]] 目前已经有方向和下一步，但仍偏方法论描述。
- 真实的压力测试场景其实已经是 [[06-项目/监管系统知识整理|监管系统知识整理]]。

判断：

- 现在最需要的不是再证明“这个想法很好”，而是定义“怎样算跑通”。
- 如果没有验证指标，项目会持续建设，但很难判断当前优化是否真的提高了系统价值。

建议：

- 为项目补几条可验证指标，例如：
  - 一个问题能否优先命中知识页，而不是回到原文全文搜。
  - 新资料进入后，是否至少更新 1 个已有知识页，而不是只新增资料页。
  - 同一问题第二次出现时，是否已经有 FAQ、规则页或综合页可以复用。

## 优先级建议

如果只做最重要的 3 件事，建议顺序如下：

1. 先补“活跃工作台”，把待处理事项显性化。
2. 再补“稳定结论升级机制”，解决高价值输出沉不下来的问题。
3. 再补“待确认与待建清单”，把开放问题集中管理起来。

## 暂时不建议优先做的事

- 不建议现在优先上更复杂的检索栈。
- 不建议现在优先做大规模自动化写回。

原因：

- 从当前仓库状态看，最大的瓶颈不是“找不到内容”，而是“知识闭环还没有完全成型”。
- 这也更符合 [[01-资料库/2026-04-05-Karpathy-How I use LLMs|Karpathy: How I use LLMs]] 的原始强调点：先把 Wiki 维护机制建立起来，再考虑更重的技术层。
