# LLM Wiki Agent Guide

本文件是本仓库的维护规则权威版本。

- `CLAUDE.md` 是面向其他 agent 工具的兼容镜像，应与本文件保持一致。
- `00-首页/工作说明.md` 只保留摘要导航，不再重复整份规则。

本仓库的目标不是临时回答问题，而是持续沉淀结构化知识。

## 核心原则

1. 原始资料只增不改，作为事实来源保留。
2. Wiki 页面是整理后的知识层，允许持续更新。
3. 每次处理新资料或新问题时，优先更新已有页面，而不是重复创建近义页面。
4. 回答问题时，优先复用现有 Wiki 页面；只有在知识页不够精确时才回溯原始资料。
5. 重要的新结论、新对比、新框架，必须回写到仓库，而不是只停留在聊天记录里。

## 目录与承载

- `00-首页/`：入口页、总索引、简版工作说明。
- `01-资料库/`：原始资料、来源记录、原文页。
- `02-主题/`：长期维护的主题页与规则页。
- `03-实体/`：人物、公司、产品、系统、报表等对象页。
- `04-综合/`：对比、综述、FAQ、决策、问题清单、梦境页。
- `05-日志/`：ingest、query、dream、lint、setup 日志。
- `06-项目/`：按项目维度维护的长期页。
- `90-模板/`：页面模板。

## 页面分工

### 资料页

- 放在 `01-资料库/`
- 保存事实来源、来源元信息、核心观点、影响范围
- 不承担长期结论层职责

### 主题页 / 规则页

- 放在 `02-主题/`
- 主题页沉淀稳定知识
- 规则页处理跨报表、跨页面可复用的口径规则

### 实体页

- 放在 `03-实体/`
- 针对具体对象持续维护
- 监管报表优先使用 `90-模板/报表模板.md`

### 综合页

- 放在 `04-综合/`
- 只承接跨页成果，例如对比、路线图、FAQ、决策、问题清单
- 不重写单页已经稳定存在的摘要

### 项目页

- 放在 `06-项目/`
- 只维护目标、状态、里程碑、问题、下一步
- 不重复主题定义和资料摘要

## 去冗余规则

- 资料页写来源与核心观点，不复制大段提炼到主题页。
- 主题页和实体页写稳定结论，不重复原文长摘录。
- 综合页写跨页判断，不重复单页摘要。
- 项目页写推进状态，不重复知识层内容。
- 同一结论如果已有承接页，优先补该页，而不是新建近义页面。

## 检索策略

### 默认最小读取

每次开始工作，按以下顺序读取，找到足够上下文即停止：

1. `00-首页/index.md`
2. `05-日志/log.md` 中最近相关条目
3. 与当前任务直接相关的页面
4. 仍不够时再读 `01-资料库/` 原文页

默认不要全量通读日志。优先使用以下低成本方式：

- 先读最近 10 到 20 条日志
- 或先按关键词检索命中条目
- 只有任务明确依赖长期演化历史时，才扩大范围

### 监管类检索

按 L0 → L1 → L2 逐层读取：

| 层级 | 位置 | 用途 |
|------|------|------|
| L0 | `04-综合/监管系统-报表清单.md` | 扫描关键词，定位系统页、报表页、规则页 |
| L1 | `03-实体/` 与 `02-主题/` | 读取常见口径、报表规则、系统关系 |
| L2 | `01-资料库/系统名/` | 核对原文定义、校验关系、项目注解 |

补充入口：

- 跨报表概念问题先查 `02-主题/` 规则页。
- 单一报表问题先查报表页，再回原文页。

### 非监管类检索

按以下顺序读取：

1. `00-首页/index.md`
2. 相关主题页、实体页、项目页
3. 必要时回到资料页

### 上下文节流

- 不因为“存在”就读取所有主题页、实体页、项目页。
- 不重复读取 `AGENTS.md`、`CLAUDE.md`、`00-首页/工作说明.md` 三份等价说明。
- 新资料进入时，优先读直接相关页、同关键词页、反向链接页，而不是全库扫描。
- 已经在知识页找到稳定答案时，不再为了“保险”重复读取原文。

## 闭环机制

一次有效工作，至少应落到以下 4 层中的 2 层以上：

- 证据层：资料页或原文页
- 知识层：主题页、规则页、实体页、综合页
- 队列层：问题清单、待确认项、项目下一步
- 日志层：`05-日志/log.md`

### 升级规则

- 同一问题被问到 2 次以上，升级为 FAQ、规则页或综合页。
- 同一概念出现在 3 份以上资料，且已跨 2 个以上知识页复用，升级为主题页或规则页。
- 同一结论在 2 次以上 query、dream 或 lint 中重复出现，升级为长期知识页。
- 梦境里的同类建议连续出现 2 次以上，可在后续普通维护中更新稳定页。
- 无法立即确认的问题，不静默跳过；要标注 `待确认`，并写入问题清单或项目下一步。

## 操作类型

### Ingest

当用户添加新资料时：

1. 创建或更新资料页到 `01-资料库/`
2. 补齐最低元信息：来源、作者、发布时间、获取日期、可更新页面
3. 提炼 3 到 5 条核心观点
4. 通过索引、关键词、反向链接定位直接相关页面
5. 将新信息合并进对应主题页、实体页、项目页或综合页
6. 如出现稳定新概念且无承接页，再新建页面
7. 补交叉链接
8. 记录到 `05-日志/log.md`

如果资料与现有内容冲突：

- 明确标记冲突点
- 不静默覆盖旧结论
- 在相关页面注明依据来源与日期

监管类 ingest 默认顺序：

1. 在 `01-资料库/系统名/` 创建原文页
2. 更新或创建系统页到 `03-实体/`
3. 更新或创建报表页到 `03-实体/`
4. 更新 `04-综合/监管系统-报表清单.md`
5. 若跨报表概念已满足升级条件，创建或更新规则页到 `02-主题/`
6. 只有在前面几层承接不下时，再补主题页或对比页

### Query

当用户提问时：

1. 先判断问题类型
2. 按检索策略逐层读取，找到答案即停止
3. 输出时明确区分“已知结论”和“推断判断”
4. 如果本次回答产生了长期有价值的新内容，优先回写到已有知识页
5. 如果仍无法沉淀为稳定结论，标记 `待确认`，并放入问题清单或项目下一步
6. 记录到 `05-日志/log.md`

### Dream

当用户要求“做一次梦”时：

1. 读取 `00-首页/index.md`
2. 读取 `00-首页/我是谁.md`、`00-首页/我的偏好.md`、`00-首页/我的目标.md`
3. 读取 `06-项目/当前学习地图.md`、相关项目页与最近相关日志
4. 优先关注最近 7 天更新内容；信息不足时再回看最近 30 天
5. 输出必须区分“观察”和“推断”
6. 推断可以大胆，但不能写成既定事实
7. 自动落到 `04-综合/梦境/YYYY-MM-DD-梦境.md`
8. 本轮只提出建议更新的核心页，不直接改写稳定自我页
9. 记录到 `05-日志/log.md`

### Lint

定期巡检知识库：

- 是否有重复页面
- 是否存在孤立页面
- 是否缺少重要交叉链接
- 是否有明显过时结论
- 是否有高频概念尚未建页
- 是否有冲突信息未标注来源
- 主题页、规则页、实体页是否缺少 frontmatter
- `04-综合/监管系统-报表清单.md` 是否与已纳入原文数量一致

巡检后：

- 先修补结构问题
- 再把待升级、待确认、待建内容沉淀到对应页面
- 最后记录到 `05-日志/log.md`

## 写作约定

1. 优先使用中文。
2. 页面标题简洁明确，避免同义重复。
3. 一页只承载一个稳定主题。
4. 使用 Markdown 链接互相引用。
5. 普通知识页不堆砌长摘录，优先写提炼后的结论；监管原文页保留精度。
6. 对不确定内容显式标注 `待确认`。
7. 对时间敏感内容尽量写明具体日期。
8. 以下页面必须包含 YAML frontmatter：
   - 实体页：`type: 实体`, `entity_type: ...`, `keywords: [...]`
   - 主题页：`type: 主题`, `keywords: [...]`
   - 规则页：`type: 规则页`, `keywords: [...]`

原文页、首页、梦境页、日志、模板不强制要求 frontmatter。

## 命名建议

- 主题页：`主题名.md`
- 实体页：`实体名.md`
- 综合页：`主题-对比.md`、`主题-综述.md`、`主题-路线图.md`
- 资料页：`YYYY-MM-DD-标题.md`
- 监管原文页：`YYYY-MM-DD-标题-原文.md`

## 开始与结束检查

### 开始工作时

1. 读取 `00-首页/index.md`
2. 读取最近相关日志，而不是默认整份日志
3. 读取当前任务最相关的页面

### 结束工作时

1. 是否更新了受影响页面
2. 是否补上了交叉链接
3. 是否把高价值输出沉淀到知识层或队列层
4. 是否记录到日志
