# 巡检 / Lint 工作流

当用户要求巡检、检查、lint、死链、一致性或状态核对时，执行本流程。

## 检查范围

- `AGENTS.md`、`.agents/`、`索引.md`、`日志.md`
- `报表业务口径/`
- `概念/`
- `数据表/`
- `血缘/`
- `sql/`
- `模板/`

## 核心检查

- 旧目录残留：不得出现已废弃的多层知识目录、历史材料目录、旧 SQL 工作区或旧 SQL 子目录。
- 死链：不得保留指向已删除来源对象的 wikilink。
- 待补引用：页面中出现“待补充”“未找到”“待补（目标页名）”等引用入口时，必须按所属系统目录复查 `报表业务口径/`、`数据表/`、`血缘/`、`sql/`；如果目标对象已存在，应直接补为 Obsidian wikilink。
- 未完成检索标记：引用入口只写“检索词”但没有写候选目录或候选路径的，视为未完成检索；巡检时必须重新查找并能补则补。
- Frontmatter：知识页必须具备仓库要求的基础 frontmatter。
- 模块边界：报表页讲业务口径，数据表页讲字段和码值，血缘页讲来源去向，概念页讲复用规则。
- 系统边界：除 `概念/` 外，报表、数据表、血缘、SQL 均按系统目录分层。
- 日志：实质修复后必须追加 `日志.md`。

## 处理原则

- 能确定的结构问题直接修。
- 能通过同系统目录、表编码、中文名、SQL 文件名或页面标题定位的待补引用，直接改成 Obsidian wikilink；不得只报告“疑似可补”。
- 不能确认的业务问题写入 `Open Questions` 或日志。
- 不制造不存在页面的占位 wikilink。
- 修复后重新运行检查并记录结果。
