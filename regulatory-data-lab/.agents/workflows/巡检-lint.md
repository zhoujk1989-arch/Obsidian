# 巡检 / Lint 工作流

当用户要求巡检、检查、lint、死链、一致性或状态核对时，执行本流程。

## 检查范围

- `AGENTS.md`、`.agents/`、`索引.md`、`日志.md`
- `报表业务口径/`
- `概念/`
- `数据表/`
- `血缘/`
- `sql/`
- `模板/`

## 核心检查

- 旧目录残留：不得出现已废弃的多层知识目录、历史材料目录、旧 SQL 工作区或旧 SQL 子目录。
- 死链：不得保留指向已删除来源对象的 wikilink。
- 待补引用：页面中出现“待补充”“未找到”“待补（目标页名）”等引用入口时，必须按所属系统目录复查 `报表业务口径/`、`数据表/`、`血缘/`、`sql/`；如果目标对象已存在，应直接补为 Obsidian wikilink。
- 未完成检索标记：引用入口只写“检索词”但没有写候选目录或候选路径的，视为未完成检索；巡检时必须重新查找并能补则补。
- Frontmatter：知识页必须具备仓库要求的基础 frontmatter。
- 模块边界：报表页讲业务口径，数据表页讲字段和码值，血缘页讲来源去向、字段落地状态和上下游对象，概念页讲系统入口和跨对象复用规则。
- 血缘页链接完整性：血缘页中的直接上游对象、直接下游对象、Nodes、表级 Edge List、字段级 Edge List、关键过滤与依赖条件、回链检查，如果对象或 SQL 文件已存在，必须使用 Obsidian wikilink；不得保留裸对象名、裸路径或仅用反引号包裹的内部对象。
- 表格 wikilink 格式：Markdown 表格内的 SQL 文件 wikilink 使用别名时，竖线必须写成 `\|`；巡检时发现未转义别名导致表格列数异常的，应修成转义 wikilink 或无别名 wikilink。
- 数据表越界内容：数据表页不得保留“字段落地状态”“SQL 是否已明确产出”“直接上游对象”“直接下游对象”“上下游对象”等章节；巡检时发现应迁移或标记到血缘页，并在数据表页只保留血缘页引用入口。
- 概念页越界内容：概念页不得承载单张报表完整口径、单张数据表完整字段字典、单条血缘字段级边、SQL 产出状态、上下游对象、长 SQL、DDL 全文或监管原文；巡检时发现应移回对应对象页，概念页只保留摘要和 Obsidian wikilink 入口。
- 系统边界：除 `概念/` 外，报表、数据表、血缘、SQL 均按系统目录分层。
- 日志：实质修复后必须追加 `日志.md`。

## 处理原则

- 能确定的结构问题直接修。
- 不使用脚本/正则/批量工具对目录进行机械扫描来判定引用是否存在或缺失。发现疑似缺失时，先逐篇手动阅读文件内容，理解其业务含义、上下游关系和上下文语境，确认目标页面是否确实存在且应当被引用后，再决定是否补链。
- 能通过同系统目录、表编码、中文名、SQL 文件名或页面标题定位的待补引用，直接改成 Obsidian wikilink；不得只报告"疑似可补"。
- 血缘页 Edge List 里的对象名应逐项核对：已存在对象补 wikilink；不能确认关联的，不补链并写入待确认说明。
- 不能确认的业务问题写入 `Open Questions` 或日志。
- 不制造不存在页面的占位 wikilink。宁可在引用入口写「待补充（未找到：候选路径；检索词：...）」，也不凭空构造指向不存在页面的 wikilink。
- 修复后重新运行检查并记录结果。
