# 血缘维护规则

血缘边统一维护在血缘页：

- 血缘页记录完整表级边、字段级边、上下游对象、字段落地状态、SQL 产出状态和证据状态。
- 源数据表页和目标数据表页不再记录"我到去哪"或"我从谁来"。
- 数据表页只保留对应血缘页的 Obsidian wikilink 作为入口。

## 可写入依据

只有 SQL、过程、视图、字段字典、用户确认材料、外部 wikilink 或既有知识页能证明的边，才能写为确认血缘；证据不足时写「待确认」或「疑似」，不得硬造。

## 引用完整性原则

- **不凭空构造链接**：任何 `[[血缘-...]]`、`[[报表-...]]`、`[[sql/...]]` 引用的增补，必须基于目标页面已实际存在于仓库中，且在业务逻辑上与当前对象确实存在关联。
- **边表对象必须可点击**：血缘页中的直接上下游对象、Nodes、表级 Edge List 的 `From`/`To`/`Evidence`、字段级 Edge List 的 `源对象`/`目标对象`，只要目标知识页或 SQL 文件已存在，就必须写成 Obsidian wikilink；不得只写裸对象名、裸路径或反引号路径。
- **表格内 SQL 链接必须转义竖线**：Markdown 表格单元格内的 SQL wikilink 如使用别名，写作 `[[sql/<系统名>/<文件名>.sql\|<文件名>.sql]]`，避免 `|` 被当成表格分隔符。
- **不依赖程序脚本直接匹配**：巡检或修复过程中，不使用脚本/正则/批量工具对目录进行机械扫描来判定引用是否存在或缺失。如发现疑似缺失，应先手动阅读文件内容，理解其业务含义、上下游关系和上下文语境，确认目标页面是否确实存在且应当被引用，在充分理解的基础上再做修复决策。
- 宁可写「待补充（未找到：目标路径；检索词：...）」，也不构造一个指向不存在页面的 wikilink。

## 更新顺序

1. 更新或创建血缘页，记录表级边、字段级边、上下游对象、字段落地状态、处理方式和证据。
2. 确认相关数据表页"引用入口"存在对应血缘页 wikilink；如缺失则补链。
3. 如涉及报表业务口径变化，同步更新报表业务口径页。
4. 如涉及系统级规则或通用概念，同步更新系统页或概念页。
5. 追加 `日志.md`。

## 最低字段级血缘

对监管报表、标准接口表、核心下游结果表，字段级血缘至少覆盖所有目标输出字段。

每个目标字段必须说明：

- 来源对象
- 来源字段
- 处理逻辑
- 关系类型
- 证据状态

无来源字段写「待补血缘」或「无映射来源，待确认」，不得写成已闭环。
