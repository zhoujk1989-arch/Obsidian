# 血缘双向维护规则

当 ingest SQL、创建或更新血缘页、补充数据表上下游、做血缘影响分析或 SQL 开发时，必须执行本规则。

目标是让每张核心数据表都能回答两个问题：

- 我从谁来：本表字段、记录或指标的直接上游来源是什么。
- 我到谁去：本表字段、记录或指标被哪些下游表、报表、接口、SQL 或校验消费。

## 维护原则

- 血缘边必须双向落账：在血缘页记录完整边，在源数据表页记录下游消费，在目标数据表页记录上游来源。
- 数据表页只维护摘要级上下游，不展开完整字段级血缘；完整字段级边仍维护在血缘页。
- 上游表页的“直接下游对象”必须能看出本表去向；目标表页的“直接上游对象”必须能看出本表来源。
- 跨系统血缘按主目标系统存放血缘页，同时同步更新涉及系统下的相关数据表页。
- 只有 SQL、过程、视图、字段字典、来源页或既有知识页能证明的边，才能写为确认血缘；证据不足时写“待确认”或“疑似”，不得硬造。
- 不创建指向不存在页面的占位 wikilink；待建页面写普通文本“待补（页面名）”。

## 需要同步维护的对象

每新增或修改一条表级或字段级血缘边，至少检查：

1. 血缘页：记录表级边、字段级边、处理逻辑、关系类型和证据。
2. 源数据表页：在“直接下游对象”记录目标表、报表、SQL、血缘页和被消费字段摘要。
3. 目标数据表页：在“直接上游对象”记录来源表、来源字段摘要、处理方式和血缘页。
4. 报表业务口径页：如目标是报表或监管接口，记录字段来源摘要并链接血缘页。
5. 来源页：如本次新增证据来自 SQL、DDL、字段字典或外部 wikilink，记录关键发现。
6. 系统页：如新增报表、表族、血缘页或跨系统关系，更新入口或覆盖摘要。
7. `日志.md`：记录本次双向维护范围、状态判断和遗留问题。

## 数据表页写法

### 直接上游对象

目标表页应写清本表来自哪里，建议使用表格：

| 上游对象 | 上游字段/范围 | 处理方式 | 证据/血缘页 | 确认状态 |
| --- | --- | --- | --- | --- |
| 数据表-... | `FIELD_A`, `FIELD_B` | 直接映射 / 关联 / 过滤 / 聚合 / 码值转换 | 血缘-... | 确认 / 待确认 |

### 直接下游对象

源表页应写清本表去到哪里，建议使用表格：

| 下游对象 | 消费字段/范围 | 用途 | 证据/血缘页 | 确认状态 |
| --- | --- | --- | --- | --- |
| 数据表-... / 报表-... / SQL-... | `FIELD_A`, `FIELD_B` | 生成目标字段 / 过滤 / 关联 / 校验 | 血缘-... | 确认 / 待确认 |

## 血缘页写法

血缘页仍是完整技术血缘的主承载页。除表级边和字段级边外，必须保留：

- 直接上游对象
- 直接下游对象
- 相关数据表页
- 相关报表业务口径页
- 回链检查

回链检查应明确列出：

- 源数据表页是否已写入下游消费。
- 目标数据表页是否已写入上游来源。
- 报表业务口径页是否已回链血缘页。
- 跨系统数据表页是否已同步更新。

## Ingest 执行要求

### SQL ingest

SQL ingest 解析出 `源对象 -> 目标对象` 后，不能只更新目标报表、目标表和血缘页，还必须逐一回写所有确认上游数据表页的“直接下游对象”。

如果某个上游数据表页尚不存在：

- 有表结构或字段字典证据时，按数据表 ingest 规则创建数据表页。
- 没有表结构证据时，不强行创建完整数据表页，在血缘页和目标表页记录普通文本“待补（数据表-...）”，并写入 `Open Questions`。

### 数据表 ingest

数据表 ingest 只拿到表结构时，不硬造血缘；但如果已有血缘页或 SQL 证据引用该表，必须同步补齐该表页的上游或下游摘要。

### SQL 开发

agent 生成 SQL 草案时，草案阶段也要维护草案血缘：

- 目标表页记录“SQL 草案加工来源”，状态为待验证。
- 源表页记录“被 SQL 草案消费”，状态为待验证。
- 血缘页标注证据为工作区 SQL 草案，不写成生产已验证。

## 影响分析要求

做字段或表影响分析时，不得只从被变更表的下游列表出发。必须同时读取：

- 被变更表页的直接上游对象，用于判断字段来源是否受影响。
- 被变更表页的直接下游对象，用于判断影响去向。
- 相关血缘页的字段级边，用于确认字段如何传递。
- 下游对象的数据表页或报表页，用于继续追踪间接影响。

## 完成标准

一次血缘维护只有同时满足以下条件，才视为完成：

- 血缘页中有完整表级边和必要字段级边。
- 源数据表页能看到本表被谁消费。
- 目标数据表页能看到本表从谁加工而来。
- 相关报表页、来源页、系统页按需更新。
- 回链检查无已知缺口；如有缺口，已写入 `Open Questions`。
- `日志.md` 已记录本次触达对象和未确认问题。
