# 概念-索引体系与检核方法

## 定义

索引不是知识正文的副本，而是让人和 agent 以最短路径进入正确知识对象的路由层。根索引只承载系统和模块，不承载具体报表清单。

regulatory-data-lab 的索引采用四层结构：

1. 根索引：回答“现在该从哪里进入”。
2. 系统入口：回答“这个对象属于哪个系统和主题域”。
3. 对象入口：回答“这是报表、数据表、来源还是血缘”。
4. 检核入口：回答“本次工作是否缺证据、缺互链、缺状态判断”。

## 读取路由

| 问题类型 | 第一入口 | 第二入口 | 停止条件 |
| --- | --- | --- | --- |
| 报表业务口径 | 系统页中的报表业务口径入口 | 报表页、来源页或数据表页 | 已能说明报送对象、范围、纳入排除规则和字段业务含义 |
| 字段含义或码值 | 系统页中的表族入口 | 数据表页、来源页或相关报表页 | 已能说明字段含义、取值、码值、落地状态 |
| 字段级来源 | 系统页中的技术追溯入口 | 血缘页、数据表页、来源页、SQL | 已能说明源字段、加工逻辑和缺口字段 |
| 证据核对 | 系统页中的来源定位方式 | 来源页、外部 `regulatory-knowledge-vault` wikilink、本地 SQL/DDL | 已能定位本地证据和外部来源 |
| SQL 开发 | 系统页中的相关报表/表族入口 | 数据表页、报表页、来源页、血缘页、工作区 SQL 草案 | 已能形成可追溯字段映射和验证 SQL |

## 对象边界

| 对象 | 索引中应出现 | 索引中不应出现 |
| --- | --- | --- |
| 报表业务口径页 | 系统页中的报表级入口、业务用途、深挖条件 | 根索引中的全量报表清单、完整字段级血缘 |
| 数据表页 | 系统页中的表族入口、代表页、字段主档定位方式 | 根索引中的全量数据表清单、每张表的完整字段字典 |
| 来源页 | 系统页中的证据包定位方式、代表页 | 根索引中的全量来源页清单、大段监管原文或 SQL 片段 |
| 血缘页 | 系统页中的技术追溯入口 | 根索引中的全量血缘清单、业务问题默认入口 |
| 概念页/系统页 | 通用规则和系统边界入口 | 单张报表的字段明细 |

## 层级维护规则

- 根索引只承载系统总览、项目模块说明、任务路由和检核清单。
- 具体报表、数据表、来源页和血缘页全部下沉到系统页或系统级子索引。
- 大批量数据表和来源页按系统页或文件路径定位，不在根索引重复铺开。
- 血缘页只在系统页的“技术追溯入口”集中列出，避免 agent 查业务口径时默认遍历。
- 系统页按主题域或编号段组织，例如一表通 `T_1` 至 `T_10`、EAST5.0 `IE_001` 至 `IE_010`。
- 每次新增对象后，先判断是否已有相同系统和相同业务对象的页面；优先更新已有页。
- 不创建指向不存在页面的占位 wikilink；待补对象写成普通文本。

## 检核方法

### 任务路由检核

- 业务问题是否从报表业务口径页进入。
- 字段问题是否从数据表页进入。
- 加工来源问题是否从血缘页进入。
- 证据问题是否从来源页进入。
- SQL 开发是否先读数据表页、报表页、来源页，再读血缘页。

### 完整度检核

- 报表页是否说明整表报送范围、纳入排除规则、时间口径和字段/指标业务摘要。
- 数据表页是否说明表角色、主键、字段口径、字段落地状态、码值和上下游。
- 来源页是否说明本地材料、外部 wikilink、检索关键词、关键发现和 Open Questions。
- 血缘页是否说明表级链路、字段级来源、加工逻辑、缺口字段和影响。

### 状态检核

- 只有 DDL 或字段字典时，通常保持 `draft`。
- 只有 SQL 草案且未运行验证时，通常保持 `draft`。
- 外部监管来源、字段口径、互链、索引和日志都闭环后，才评估 `validated`。
- 新来源削弱旧结论时，相关页面应降回 `draft` 并写明冲突。

### 索引检核

- 根索引不得重新堆叠具体报表、数据表、来源页和血缘页。
- 根索引中的每个 wikilink 必须指向已存在页面。
- 新增报表业务口径页后，应补入对应系统页的报表入口；根索引只在系统摘要需要变化时更新计数或状态。
- 新增系统或主题域后，应补入根索引系统面板和对应系统页。
- 批量新增数据表时，应更新对应系统页的表族计数和代表页，不在根索引逐表平铺。
