# 报表业务需求目录说明

## 目录职责

本目录保存监管报表的业务需求加工规则页，是 SQL 与业务自然语言之间的转换层。

页面核心职责：

- 把对应 SQL 的加工逻辑翻译成业务人员可以评审的自然语言。
- 把业务自然语言需求沉淀成可反向生成 SQL 的加工规则。
- 只表达加工规则，不承载字段或指标的业务含义。

## 内容边界

应放入：

- 整表加工目标和报送模式。
- 来源选择、表间关联、过滤条件和去重规则。
- 目标字段或指标对应的加工规则。
- 码值转换、日期转换、金额转换、拼接、截取、默认值和空值处理规则。
- 规则优先级、例外场景、终态纳入或排除规则。
- 反向生成 SQL 所必需的条件、口径边界和待确认点。
- 对应 SQL 文件引用。
- `regulatory-knowledge-vault` 中对应报表业务实体 wikilink。

不应放入：

- 字段或指标的业务含义解释。
- 完整字段字典。
- 完整字段级血缘。
- 数据表页、血缘页、系统页等本仓库其他知识页的正文引用。
- SQL、存储过程全文或跑批脚本全文。
- 大段监管原文。
- 没有实际页面的占位 wikilink。

## 分层规则

- 必须按系统分目录保存：`报表业务口径/<系统名>/`。
- 新系统先建立或更新 `概念/概念-系统-<系统名>.md`，再新增系统目录。
- 跨系统报表放在主报送系统目录下，并在正文说明涉及的其他系统。

## 命名规则

- 页面文件名使用 `报表-<报表名称或编码>.md`。
- 标题优先使用报表名称；技术编码写入 frontmatter 或专门章节。

## 联动要求

新增或大幅更新报表业务需求页时：

- 页面正文只引用对应 SQL 文件和外部报表业务实体。
- 需要在 `日志.md` 记录本次规则调整、状态判断和遗留问题。
- 如加工规则影响数据表或血缘维护，agent 可在内部同步检查，但不把这些内部检查链接写成本页面正文引用。
