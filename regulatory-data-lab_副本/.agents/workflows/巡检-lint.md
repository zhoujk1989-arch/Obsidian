# 巡检/Lint 工作流

lint 不是泛泛“看一眼”，而是一种可执行维护动作。对于可直接修复的问题，agent 应直接修复；无法修复的问题必须写入 `Open Questions`、必要时降级状态，并记录日志。

## 默认输入

- `索引.md`
- `日志.md`
- 最近修改页
- 相关来源页、报表业务口径页、数据表页、血缘页、概念页
- 相关 SQL 草案、正式 SQL、存储过程或视图

巡检应优先按“对象包”检查，而不是只逐页检查。一个对象包通常包括：

- 来源页
- 报表业务口径页
- 数据表页（目标表、核心源表、中间表或稳定逻辑数据集）
- 血缘页
- SQL 草案、正式 SQL、存储过程或视图（如有）
- 相关系统页或概念页

## 默认检查项

- 孤儿页面或几乎没有入链的页面
- 缺少上下游链接的数据表页
- 数据表页上下游是否符合 `.agents/rules/血缘双向维护规则.md`：源表页有下游消费，目标表页有上游来源
- 血缘页中提到、但还没有沉淀到表页或概念页的关键对象或规则
- 业务定义与 SQL 过滤条件是否矛盾
- 是否存在被新 SQL 或新规则替代的陈旧页面
- 来源页是否缺少 `external_paths` 字段，或在未定位外部来源时没有记录待检索缺口
- 是否存在指向不存在页面的 wikilink
- 是否仍有“待补充”出现在必填章节
- `validated` 页面是否仍满足验证条件
- 接口表中的代码类字段是否已写清码值说明
- 根索引是否被具体对象清单污染
- 如果本次涉及表、字段、SQL 逻辑或口径变更，是否已按 `.agents/rules/血缘影响分析规则.md` 做影响分析

## 四页关系巡检

### 来源页

- 来源页必须能解释当前结论的证据来源：本地 SQL/DDL/字段字典、外部 `regulatory-knowledge-vault` wikilink、检索关键词、关键发现和冲突。
- 来源页中的本地文件路径必须真实存在；外部来源必须只写 Obsidian wikilink，不写本机绝对路径。
- 如果报表业务口径页、数据表页或血缘页引用了某个监管规则、码值、字段要求或 SQL 逻辑，必须能回到来源页或本地材料。
- 如果 `external_paths` 为空，必须有待检索关键词或 `Open Questions` 说明缺口。

### 报表业务口径页

- 必须说明整表报送范围、统计/明细粒度、纳入排除规则、时间口径、字段或指标口径。
- 关键规则必须能追溯到来源页、SQL、数据表页或既有知识页。
- 不得只复述 SQL；但业务规则必须能与 SQL 的 `where`、`join`、`case when`、聚合和常量字段逐项对账。
- 如果 SQL 中存在实质性过滤条件，而报表业务口径页没有业务解释，应视为遗漏。
- 如果报表业务口径页声称纳入或排除某类对象，但 SQL 没有对应实现或来源证据，应写入 `Open Questions`，必要时降级状态。

### 数据表页

- 必须覆盖字段口径、码值、字段落地状态、上下游对象和相关来源/血缘链接。
- 字段口径完整表默认在数据表页维护；报表业务口径页和血缘页只保留摘要或技术来源。
- 必须区分目标表加工状态和源表下游使用状态。
- 代码类字段必须写清码值、中文含义、映射规则、特殊值、证据来源；不得只写“见码表”。
- 如果血缘页使用了某个表或字段，但数据表页没有字段口径或表角色说明，应视为遗漏。
- 如果血缘页已确认 `源对象 -> 目标对象`，但源数据表页没有记录下游消费，或目标数据表页没有记录上游来源，应视为双向维护缺口。

### 血缘页

- 必须覆盖表级链路，并说明源表、目标表、中间层、过程或脚本之间的关系。
- 对监管报表、标准接口表、核心下游结果表，字段级血缘必须至少覆盖所有目标输出字段。
- 每个目标字段都应说明来源对象、来源字段、处理逻辑、关系类型和证据。
- 无 SQL 来源的字段不能标为已闭环。
- 血缘页中的字段级来源必须能与 SQL 的 `select` 目标字段、`insert` 字段顺序、`case when`、聚合、窗口函数和常量逐项对上。
- 如果血缘页列出的上游/下游对象没有在对应数据表页回链，应修复互链或写明缺口。
- 血缘页回链检查必须覆盖源数据表页下游消费、目标数据表页上游来源和跨系统数据表页同步情况。

### SQL 与知识页

- SQL 的目标表、目标字段顺序、主键/唯一性、刷新方式必须与目标数据表页和实现说明一致。
- SQL 的过滤条件、关联条件、时间口径、码值转换、常量字段和聚合规则必须与报表业务口径页一致。
- SQL 中实际读取的源表和字段必须在血缘页或数据表页中体现。
- SQL 草案未运行或未确认投产前，不得把相关页面状态提升为 `validated`。
- 如果 SQL 与来源页、报表业务口径页、数据表页或血缘页冲突，应优先记录冲突，不得静默覆盖旧结论。

## 死链规则

- 知识页中的不存在 wikilink 视为错误。
- `日志.md`、`AGENTS.md`、`模板/` 中作为历史记录或示例出现的占位链接，不纳入知识图完整性判断。
- 如果对象尚未建立，使用普通文本 `待补充` 或 `待补（目标页名）`，不要制造空链接。

## 处理规则

- 可直接修复的问题直接修复。
- 暂时无法修复的问题写入相关页面的 `Open Questions`。
- 必要时降级页面状态。
- 如果修了互链、状态、系统页或共享知识页，也要同步更新这些对象。
- 具体对象入口应更新到系统页；根索引只在系统摘要或模块规则变化时更新。
- 表、字段、SQL 逻辑或口径变更引发的 lint，必须输出影响分析摘要、影响明细、建议动作和验证方案。
- `日志.md` 追加 `audit` 或 `lint` 条目。

## 输出要求

- 如果本次 lint 没有改文件，也要给出检查范围、未发现问题或剩余风险。
- 如果本次 lint 改了文件，至少说明被修复页面、状态变化、日志更新和仍存风险。
