# WisdomLedger

这个仓库把“监管报表 SQL”沉淀成一个可持续维护的 Obsidian 知识库。目标不是一次性问答，而是把 SQL、业务口径、血缘关系编译成一层持续积累的 wiki。

这个仓库不存储报表和指标发文的原始内容。监管原文、填报说明、报表页和主题页统一从外部知识库 `/Users/zhoujingkun/Documents/GitHub/Obsidian/my-dev-brain` 关联和检索。

## 设计思路

参考 Karpathy 的 `LLM Wiki` 思路，这个库分成三层：

- `原始材料/`：原始材料，只读。这里只放 SQL 和本地技术材料，不放监管发文原文。
- `知识库/`：LLM 维护的知识层。放报表页、指标页、表页、字段页、血缘页。
- `AGENTS.md`：给 LLM 的操作规约，约束 ingest、查询、回写、巡检方式。

这比传统 RAG 更适合监管报表场景，因为监管 SQL 的理解不是“每次现查现拼”，而是需要长期沉淀：

- 同一个指标会在多个报表中重复出现
- 同一张源表会支撑多条监管链路
- 口径变化需要留下历史和影响范围
- SQL 技术实现和业务自然语言定义需要长期对齐

## 使用方式

### 1. 放入本地 SQL

把新材料放到：

- `原始材料/SQL/`：报表 SQL、ETL SQL、视图定义

监管原文、填报说明、报表制度和主题规则不要复制进来，统一去外部知识库找：

- `/Users/zhoujingkun/Documents/GitHub/Obsidian/my-dev-brain/04-综合/监管系统-报表清单.md`
- `/Users/zhoujingkun/Documents/GitHub/Obsidian/my-dev-brain/03-实体/`
- `/Users/zhoujingkun/Documents/GitHub/Obsidian/my-dev-brain/02-主题/`
- `/Users/zhoujingkun/Documents/GitHub/Obsidian/my-dev-brain/01-资料库/`

### 2. 让 LLM 执行 ingest

可直接给 Codex 这类指令：

```text
请 ingest 原始材料/SQL/<文件名>.sql，
并到 /Users/zhoujingkun/Documents/GitHub/Obsidian/my-dev-brain 关联对应的报表原文、报表页和主题页，
把它沉淀成业务能看懂的报表口径、指标定义和血缘页面，并更新索引/日志。
```

### 3. 在 Obsidian 中浏览

重点看这些页面：

- `知识库/报表/`：监管报表是什么
- `知识库/指标/`：指标口径怎么定义
- `知识库/数据表/`：源表和中间层表是什么
- `知识库/血缘/`：链路怎么走
- `知识库/来源/`：本地 SQL 与外部监管来源如何关联

## 推荐工作流

1. 新 SQL 到仓库
2. LLM 先到 `my-dev-brain` 查对应报表清单、报表页、主题页和原文页
3. LLM 再做 SQL 结构抽取，识别源表、CTE、聚合指标、过滤条件
4. LLM 写入业务口径页和血缘页，只记录外部来源路径，不复制监管原文
5. 人工校验关键口径
6. 后续查询和追溯直接基于本库语义层 + 外部知识库原文层联合完成

## 轻量工具

`工具/血缘脚手架.py` 会先从 SQL 中提取：

- CTE 顺序
- `FROM/JOIN` 的候选源表
- 常见聚合指标

它不是完整 SQL parser，但足够生成第一版血缘草稿，减轻每次 ingest 的整理成本。

```bash
python3 工具/血缘脚手架.py 原始材料/SQL/示例-对公存款余额日报.sql
```

`工具/检索外部知识库.py` 会到 `my-dev-brain` 中查找候选报表页、主题页、原文页：

```bash
python3 工具/检索外部知识库.py "G01 资产负债项目统计表"
```

## 当前仓库附带的示例

仓库内放了一套最小示例，演示如何把一份“对公存款余额日报”的 SQL 转成：

- 报表业务定义
- 两个指标口径页
- 两张源表页
- 一个关键字段页
- 一张端到端血缘页
- 一个外部来源关联页

你后续只要继续按这个结构 ingest 新报表，wiki 就会越来越像一个监管报表知识图谱，而不是零散的 SQL 注释集。
