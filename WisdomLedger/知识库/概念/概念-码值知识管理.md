# 概念-码值知识管理

## 结论

- 不要把“码值”只存成一张静态码表，也不要只在字段页顺手写几条说明。
- 在 `LLM Wiki` 模式下，码值应拆成三层管理：原始层、知识层、映射层。
- 最稳定的主键不是“码值文本”，而是 `系统 + 码表/字段 + 码值`。
- 业务复用的核心不是“记住某个码”，而是沉淀“这个码在什么上下文下代表什么业务含义”。
- 码值知识默认是单系统知识；跨系统关系必须通过映射页显式表达。

## 推荐存储分层

### 1. 原始层

- 各系统原始码表、DDL、接口文档、SQL 取数逻辑，保留在 `原始材料/` 或外部系统文档。
- 原始层不追求统一命名，只保留事实来源。

### 2. 知识层

- 在 `知识库/数据表/` 记录哪张表、哪个字段是稳定码值承载字段。
- 在 `知识库/字段/` 记录字段的业务含义、取值域、是否为标准码、是否跨系统复用。
- 当某组码值本身是长期复用知识时，单独建立概念页或字段页中的专章，不要散落在多个报表页里。
- 每组码值知识必须先归属到某个系统，再讨论它是否需要跨系统映射。

### 3. 映射层

- 新增一类可复用对象来表达“码值集合”与“跨系统映射关系”。
- 即使当前仓库还没单独建目录，也建议先按页面方式管理，后续再决定是否独立成 `知识库/码值/`。
- 每个码值集合页面只描述一个明确上下文，例如：
  - 客户类型码
  - 账户状态码
  - 产品类型码
  - 机构类型码

## 推荐的最小知识对象

要做到长期可维护，至少分成下面三种对象。

### 1. 字段页

- 表达“哪个字段承载这个码”。
- 例如 `[[字段-客户类型代码]]` 应记录：
  - 所属表
  - 字段技术名
  - 字段业务定义
  - 是否枚举码值
  - 关联到哪个码值集合页

### 2. 码值集合页

- 表达“这一组码整体在说什么”。
- 推荐命名形式：
  - `概念-码值集合-客户类型`
  - `概念-码值集合-账户状态`
- 一页内记录：
  - 所属系统
  - 适用系统
  - 适用字段
  - 码值表格
  - 与监管口径的关系
  - 是否存在跨系统映射

### 3. 映射页

- 表达“不同系统之间如何对齐”。
- 推荐命名形式：
  - `概念-码值映射-客户类型-核心系统到CRM`
- 一页内记录：
  - 源系统码值
  - 目标系统码值
  - 统一业务概念
  - 映射规则类型：一对一、一对多、多对一、条件映射
  - 生效条件和 Open Questions

## 为什么不要只建一张总码表

- 同一个码值 `01` 在不同系统经常不是同一个意思。
- 同一个业务概念在不同系统经常对应不同码值。
- 同一个字段名在不同表里未必共享同一套枚举。
- 监管口径真正关心的是“这个码在该报表上下文下是否被纳入”，不是它长得像不像。
- 如果系统边界不清，码值知识很快会退化成一堆不可复用的备注。

所以不要把主键设计成：

- `码值`

而应设计成至少可区分上下文的键：

- `source_system`
- `table_name`
- `field_name`
- `code_value`

如果后续真要落成结构化总表，建议唯一键按这个粒度。

## 页面应该记录什么

码值知识页推荐包含以下信息：

| 字段 | 含义 |
| --- | --- |
| source_system | 来源系统 |
| source_table | 来源表 |
| source_field | 来源字段 |
| code_value | 原始码值 |
| code_label | 原始码值名称 |
| business_meaning | 业务解释 |
| normalized_concept | 归一概念 |
| valid_from / valid_to | 生效区间 |
| evidence | SQL、DDL、文档路径 |
| upstream_links | 上游来源页、字段页、数据表页 |
| downstream_links | 报表页、指标页、血缘页 |

其中最重要的是：

- `business_meaning`
- `normalized_concept`
- `evidence`

因为这三项决定它是不是知识，而不只是字典。

## 在 WisdomLedger 中的推荐落地方式

### 方案 A：轻量起步，优先推荐

- 不先新建 `知识库/码值/` 目录。
- 先在 `字段页` 里新增 `## 码值定义` 专章。
- 对复用频繁的码值集合，再单独建 `概念-码值集合-...` 页面。
- 对跨系统映射，再建 `概念-码值映射-...` 页面。
- 所有码值页都先按系统拆开维护。

这个方案最适合你当前阶段，因为：

- 结构改动小
- 与现有 `字段/数据表/指标/报表` 体系兼容
- 更符合 `LLM Wiki` 的“先沉淀知识，再决定是否抽象成 schema”

### 方案 B：规模化后独立目录

- 当码值集合很多、跨系统映射很多时，再新增：
  - `知识库/码值/`
- 目录下可分两类页：
  - `码值集-...`
  - `码值映射-...`

这个方案适合：

- 系统数很多
- 码值映射经常变
- 你准备进一步把 wiki 和结构化抽取联动

## 一条实用原则

- “系统负责定义边界”
- “字段负责承载上下文”
- “码值集合页负责表达枚举语义”
- “映射页负责表达系统间对齐规则”

这比把所有东西塞进一个 Excel 或一张码值总表更接近 `LLM Wiki` 的知识组织方式。

- 所属系统：
- 系统：
- 字段：
- 相关页面：

| --- | --- | --- | --- | --- |

## 与系统概念的关系

- 码值知识必须服从 `[[概念-系统边界]]`。
- 默认假设是“系统间彼此独立”。
- 只有在有明确证据时，才建立跨系统码值映射，而不是直接合并成统一字典。
- 码值集合页和码值映射页的文件路径也应按所属系统分层存放。

## 和 Karpathy 的 `LLM Wiki` 对齐点

- 原始码表仍留在原始层，不污染知识层。
- 知识层不复制所有原始字典，而是提炼“可复用语义单元”。
- 每个结论都能回链到字段、表、SQL 和外部文档证据。
- 当映射发生变化时，只需更新映射页，不必重写所有报表页。

## 最后的判断

- 不存在“完美”的一次性设计，只有“可持续演化”的结构。
- 对你这个仓库，最佳起点不是大而全的主数据平台，而是：
  - 先把码值挂到字段页
  - 把高复用枚举抽成码值集合页
  - 把跨系统对齐抽成映射页

这样最符合 `LLM Wiki` 的增量沉淀方式。
