# WisdomLedger Agent Schema

This vault is an LLM-maintained knowledge base for turning regulatory report SQL into:

1. business-readable natural language definitions
2. persistent lineage documentation
3. reusable report, metric, table, and field knowledge

The operating model follows the three-layer pattern from Karpathy's "LLM Wiki":

- `原始材料/`: immutable source material. SQL, regulatory excerpts, screenshots, data dictionaries, meeting notes.
- `知识库/`: the maintained knowledge layer. The LLM writes and updates this.
- `智能体规范.md`: the schema and workflow contract that keeps the vault disciplined.

## Repository Layout

- `原始材料/SQL/`: raw SQL files for reports or intermediate jobs.
- `原始材料/背景材料/`: business rules, regulatory excerpts, data dictionaries, mapping notes.
- `知识库/来源/`: one page per ingest source packet.
- `知识库/报表/`: one page per regulatory report or schedule.
- `知识库/指标/`: one page per business metric or regulatory indicator.
- `知识库/数据表/`: one page per physical table or stable logical dataset.
- `知识库/字段/`: one page per critical field that appears in metric logic.
- `知识库/血缘/`: end-to-end lineage pages.
- `知识库/概念/`: reusable domain concepts and glossary entries.
- `模板/`: page skeletons. Copy structure, do not keep placeholders in final pages.
- `索引.md`: content-oriented catalog.
- `日志.md`: append-only chronology.

## Naming Rules

- Source pages: `来源-...`
- Report pages: `报表-...`
- Metric pages: `指标-...`
- Table pages: `数据表-...`
- Field pages: `字段-...`
- Lineage pages: `血缘-...`
- Concept pages: `概念-...`

Page titles should prefer business names in Chinese. Keep technical identifiers in frontmatter or dedicated sections.

## Core Principles

1. Raw sources are immutable. Never rewrite files under `原始材料/`.
2. Business meaning comes before SQL description.
3. Every important claim should point back to evidence:
   - a raw SQL file
   - a regulatory/context document
   - an existing wiki page
4. Update existing pages instead of duplicating near-identical pages.
5. If SQL meaning is ambiguous, state uncertainty explicitly under `## Open Questions`.
6. When a user asks a valuable question that produces reusable knowledge, file the answer back into the wiki.

## Required Frontmatter

Every wiki page should include:

```yaml
---
type: report|metric|table|field|lineage|source|concept
id: 报表-...
status: draft|validated
updated: YYYY-MM-DD
tags:
  - regulatory
  - lineage
---
```

Add more keys only when they are useful and stable.

## Ingest Workflow

When a user asks to ingest a new regulatory report SQL:

1. Read the new files under `原始材料/`.
2. Create or update one `知识库/来源/来源-...` page.
3. Create or update the related `知识库/报表/报表-...` page.
4. Create or update metric pages for each business metric exposed by the SQL.
5. Create or update table pages for each physical input table and important intermediate logical dataset.
6. Create or update field pages for business-critical fields used in filters, joins, and aggregations.
7. Create or update one lineage page showing the transformation path end to end.
8. Update `索引.md`.
9. Append a dated entry to `日志.md`.

## How To Translate SQL

For each SQL artifact, extract:

- report purpose
- statistical grain
- dimensions
- measures
- inclusion rules
- exclusion rules
- temporal logic
- source tables
- join logic
- transformation stages
- final output fields

Then express them in business language:

- "What is being counted or summed?"
- "At what date/grain is it observed?"
- "Who is included and excluded?"
- "Which source tables are authoritative?"
- "Which filters materially change business meaning?"

Avoid copying SQL into prose. Paraphrase the business meaning first, then preserve the SQL expression as evidence.

## Lineage Rules

Each lineage page should include:

1. a one-paragraph business narrative
2. a node list
3. an edge list table
4. a Mermaid graph when feasible
5. known gaps or manual assumptions

Separate:

- physical lineage: table to table
- logical lineage: CTE, temp result, derived metric
- semantic lineage: regulatory indicator -> business metric -> SQL expression -> source field

## Query Workflow

When answering a question:

1. read `索引.md` first
2. open the most relevant wiki pages
3. synthesize an answer with links to the relevant pages
4. if the answer adds durable knowledge, write it back into the wiki

## Lint Workflow

Periodically check for:

- orphan pages with few or no inbound links
- metric pages with no evidence section
- table pages missing upstream or downstream links
- lineage pages that mention nodes without dedicated pages
- contradictions between business definition and SQL filters
- stale pages superseded by newer SQL or rule documents

## Writing Style

- Use concise Chinese for business explanations.
- Preserve English or snake_case only for technical identifiers.
- Prefer bullet lists and tables over long narrative blocks.
- Use Obsidian wiki links like `[[报表-对公存款余额日报]]`.
- Keep code snippets short and only where they prove a claim.

## Minimum Deliverable For Each New Report

A report ingest is not complete until the vault contains:

- one source page
- one report page
- one lineage page
- all core metric pages
- all core table pages
- updated index and log
