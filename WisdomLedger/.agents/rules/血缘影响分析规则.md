# 血缘影响分析规则

当用户提出“改表”“改字段”“字段删除/新增/改名/改类型/改口径”“血缘影响分析”“这个字段影响哪些报表/SQL/表”等请求时，必须执行本规则。

目标是基于现有血缘、数据表、报表业务口径、来源和 SQL 草案，给出足够准确、全面、可执行的影响分析，而不是只列一个粗略的下游清单。

## 触发条件

以下任一情况都触发血缘影响分析：

- 表结构变更：新增表、删除表、改表名、改主键、改分区、改唯一约束、改刷新方式。
- 字段结构变更：新增字段、删除字段、改字段名、改类型、改长度、改精度、改是否为空、改默认值。
- 字段口径变更：改码值、改含义、改映射规则、改脱敏规则、改派生逻辑、改时间口径、改统计粒度。
- SQL 逻辑变更：改 `where`、`join`、`case when`、聚合、窗口函数、常量赋值、过滤范围、删除重跑范围。
- 来源变更：监管来源、字段字典、DDL、接口说明或外部 `my-dev-brain` 来源挑战旧结论。
- 用户明确要求“影响分析”“看影响”“哪些报表受影响”“下游有哪些”。

## 必须读取

开始分析前，至少读取：

1. `AGENTS.md`
2. `索引.md`
3. 对应系统页：`知识库/概念/概念-系统-<系统名>.md`
4. 被变更对象的数据表页
5. 直接相关血缘页
6. 直接相关报表业务口径页
7. 直接相关来源页
8. 相关 SQL 草案、正式 SQL、存储过程、视图或校验 SQL

如果无法确定系统或对象，先用 `rg` 搜索表名、字段名、中文名、别名、目标字段名和来源字段名。不能只凭一个页面就下结论。

## 搜索要求

必须覆盖以下检索范围：

- `知识库/数据表/<系统名>/`
- `知识库/血缘/<系统名>/`
- `知识库/报表业务口径/<系统名>/`
- `知识库/来源/<系统名>/`
- `知识库/概念/`
- `原始材料/SQL/<系统名>/`
- `原始材料/表结构/<系统名>/`
- `工作区/SQL开发/<系统名>/`

跨系统血缘必须继续检索涉及系统。例如一表通字段影响 EAST5.0 时，应同时检索一表通系统和 EAST5.0系统。

字段级检索至少使用：

- 技术字段名，例如 `A010001`
- 目标字段名或下游字段名
- 中文字段名
- 表名、表中文名、报表编号
- SQL 中可能出现的别名或表达式片段

## 分析步骤

### 1. 明确变更对象

先写清：

- 所属系统
- 表或逻辑对象
- 字段名与中文名
- 变更类型
- 变更前
- 变更后
- 变更依据：用户说明、本地 DDL、字段字典、SQL、外部 wikilink 或既有知识页
- 生效范围和生效日期，如未知则写“待确认”

如果变更描述不完整，应先用现有材料补齐；仍无法补齐的内容写入“待确认问题”，但不能因此跳过影响分析。

### 2. 建立基线血缘

从被变更对象出发，建立当前基线：

- 表级上游和下游
- 字段级来源和去向
- 目标报表、目标接口表或结果表
- 依赖该字段的 SQL 表达式
- 依赖该字段的过滤条件、关联条件、分组条件、排序/窗口条件
- 相关校验 SQL

字段变更不能只看字段级边表；还必须检查 SQL 中是否在 `where`、`join`、`group by`、`order by`、窗口函数、去重逻辑、删除范围或校验 SQL 中使用该字段。

### 3. 追踪直接影响

直接影响至少包括：

- 直接读取该表或字段的 SQL、存储过程、视图、校验 SQL
- 字段级血缘中以该字段为源字段的目标字段
- 报表业务口径页中直接引用该字段或其中文含义的字段/指标
- 数据表页中将该字段列为上游来源、码值来源、主键、过滤条件或落地状态依据的对象
- 来源页中记录该字段为关键发现或版本冲突的对象

### 4. 追踪间接影响

对每个直接下游对象继续追踪它的下游，直到没有新的下游对象或证据不足为止。

间接影响至少覆盖：

- 下游表继续被其他报表或接口消费
- 派生字段继续派生其他字段
- 汇总指标依赖受影响字段
- 码值转换影响多个下游字段
- 主键、唯一性、去重逻辑影响行数或粒度
- 时间口径或过滤口径影响报送范围

间接影响必须区分“确认影响”和“疑似影响”。疑似影响要说明为什么无法确认，以及需要补什么证据。

### 5. 判定影响类型

每条影响至少标注一种类型：

- 结构兼容性：删除字段、改名、改类型、改长度、改精度、改非空约束导致 SQL 或落地失败。
- 业务语义：字段含义、统计口径、报送范围、纳入排除规则变化。
- 码值映射：码值新增、废止、含义变化、映射目标变化。
- 主键粒度：主键、唯一性、去重、汇总粒度变化。
- 时间口径：采集日期、交易日、会计日期、报送期、快照口径变化。
- 过滤与关联：`where`、`join`、补集、排除规则变化。
- 数据质量：必填为空、格式、长度、金额方向、日期格式、身份证/隐私字段变形风险。
- 监管报送：监管要求字段、报送字段、外部填报说明或实体页受影响。
- 文档一致性：来源页、报表页、数据表页、血缘页、系统页需要同步更新。

### 6. 判定严重级别

| 级别 | 含义 | 示例 |
| --- | --- | --- |
| P0 阻塞 | 会导致生产 SQL/报送链路无法运行，或监管必报字段无法落地 | 删除被 insert/select 直接使用的字段；目标表必填字段无来源 |
| P1 高 | SQL 可运行但业务口径、报送范围、关键指标或字段值会变化 | 码值映射变化、过滤条件变化、主键粒度变化 |
| P2 中 | 需要同步文档、校验或非核心下游，但不一定改变主报送结果 | 字段说明变化、非关键校验逻辑变化 |
| P3 低 | 暂未发现下游实质影响，或仅影响描述性文档 | 未被血缘和 SQL 使用的字段中文说明微调 |
| Unknown 待确认 | 证据不足，无法判定 | 缺 SQL、缺字段字典、缺外部来源 |

严重级别必须有证据。不能把“没搜到”直接写成“无影响”，只能写“在本次检索范围内未发现确认影响”。

## 准确性要求

影响分析必须满足：

- 同时覆盖表级和字段级血缘。
- 同时覆盖知识页和 SQL 文件。
- 同时覆盖直接影响和间接影响。
- 同时覆盖结构影响和业务口径影响。
- 对每个影响给出证据位置：页面名、SQL 文件路径、字段名或血缘页。
- 对没有证据的结论标为“疑似影响”或“待确认”，不得写成确认影响。
- 明确检索范围和未覆盖范围。
- 明确哪些页面或 SQL 需要修改。
- 明确建议先执行哪些验证 SQL 或人工核对。

禁止：

- 只列受影响页面，不说明字段级原因。
- 只查血缘页，不查 SQL 和数据表页。
- 只查直接下游，不追间接下游。
- 把 `rg` 没搜到当作绝对无影响。
- 把未运行 SQL 草案写成已验证生产影响。
- 制造指向不存在页面的 wikilink。

### 7. 验证方案

至少覆盖：

- SQL 是否能运行
- 目标行数是否异常
- 主键/唯一性是否重复
- 必填字段是否为空
- 码值是否越界
- 金额、余额、日期是否异常
- 变更前后抽样回溯是否一致或差异可解释
- 受影响报表字段是否仍满足报送口径

### 8. 待确认问题

列出阻塞分析准确性的缺口，例如缺 SQL、缺字段字典、缺外部来源、缺生产运行结果、缺用户确认。

### 9. 建议执行顺序

给出可操作的修复顺序，例如：

1. 先修 P0 SQL 编译或字段缺失问题。
2. 再修 P1 业务口径或码值映射。
3. 再更新数据表页、血缘页、报表业务口径页和来源页。
4. 最后运行校验 SQL，并更新系统页与日志。

## 知识页同步规则

如果影响分析导致实际修改，必须同步：

- 数据表页：字段口径、字段落地状态、上下游、代码类字段、Open Questions。
- 血缘页：表级链路、字段级边、处理逻辑、影响说明。
- 报表业务口径页：报送范围、字段/指标口径、纳入排除规则、时间口径。
- 来源页：新证据、外部 wikilink、关键发现、版本冲突。
- 系统页：报表入口、表族入口、技术追溯入口或当前问题。
- `日志.md`：记录影响分析结论、触达对象、状态变化和未确认问题。

根索引只在系统摘要、模块规则或任务路由变化时更新；不要把具体影响对象写入根索引。

## 状态处理

- 如果影响分析发现已验证结论被削弱，相关页面必须降回 `draft`。
- 如果只是发现疑似影响但证据不足，在相关页 `Open Questions` 中写明缺口。
- 如果确认无影响，也应说明检索范围；不要把“未发现”写成绝对无影响。
- SQL 草案未运行验证前，不得把影响修复后的页面升为 `validated`。
