# SQL/存储过程开发工作流

当用户要求写 SQL、生成 SQL、补 SQL、写存储过程、生成视图、加工脚本、调度脚本或校验 SQL 时，执行本流程。

必须优先使用 `.agents/skills/sql-dev/` 中的 SQL 开发 skill；同时遵守 `AGENTS.md` 的仓库级规则。本文件负责仓库内知识回写和交付纪律。

SQL 开发会产生新的加工链路或草案链路时，必须同时遵守 `.agents/rules/血缘双向维护规则.md`。不能只在目标数据表页维护下游或加工状态；必须同步维护血缘页、源数据表页的下游消费、目标数据表页的上游来源。

## 需求抽取

先澄清或自行抽取以下要素：

- 目标产物：查询 SQL、`insert select`、存储过程、视图、临时表加工、校验 SQL 或调度片段。
- 目标系统与目标表：必须落在明确系统目录下；跨系统时选择主系统目录并说明涉及系统。
- 业务目标：统计对象、数据粒度、时间口径、纳入规则、排除规则、输出字段、刷新方式。
- 运行约束：数据库方言、跑批日期参数、是否先删后插、是否幂等、是否需要增量处理、性能限制。

## 知识读取顺序

1. 读 `索引.md`，定位系统。
2. 读对应 `知识库/概念/概念-系统-<系统名>.md`。
3. 打开相关数据表页、报表业务口径页、来源页和必要血缘页。
4. 如果知识页缺失或字段口径不完整，读取 `原始材料/表结构/<系统名>/` 下的 DDL 或字段字典补充依据。
5. 如果需求依赖监管报送口径、字段是否必报、码值含义或制度解释，按 `.agents/rules/外部来源规则.md` 检索 `my-dev-brain`。

不能只凭字段名猜测业务口径。

## 实现口径确认

生成 SQL 前，必须在说明文件或交付说明中记录：

- 目标对象与数据粒度
- 使用的数据表及其角色
- 源表到目标表的血缘方向：哪些表是上游来源，哪些表是下游目标或消费对象
- 主连接键与时间条件
- 关键过滤条件
- 字段映射、常量、码值转换和派生字段
- 无法确认的假设和风险

目标字段必须可追溯：

- 已在数据表页或 DDL 中存在的字段，可以用于 SQL。
- 仅在业务描述中出现、但知识库和表结构中没有证据的字段，不得硬写；应放入 `Open Questions` 或实现说明的“待确认字段”。
- 码值字段必须使用数据表页、报表业务口径页或来源页中的既有码值说明；缺失时写成待确认，不得编造枚举。

## 草案落地

SQL 草案默认写入 `工作区/SQL开发/<系统名>/`。

推荐命名：

- 查询脚本：`SQL_<需求名>_草案.sql`
- 存储过程：`PROC_<需求名>_草案.sql`
- 视图定义：`VIEW_<需求名>_草案.sql`
- 校验脚本：`CHECK_<需求名>_校验.sql`
- 实现说明：`说明-<需求名>.md`

如果用户明确要求修改 `原始材料/SQL/<系统名>/` 中的既有 SQL，可以按用户要求修改原始 SQL，但必须在来源页和日志中记录这次人工确认的改写，不得静默覆盖历史语义。

## SQL 文件结构

SQL 文件应尽量使用 `模板/SQL开发模板.sql`，至少包含：

- 文件头：业务目标、依赖知识页、源表、目标表、参数、运行方式、未确认点。
- 参数区：跑批日期、机构范围、重跑范围等。
- 清理区：如需要先删后插，删除条件必须与插入范围一致。
- 主加工区：默认使用直接 `select` + `left join` 组织；只有多阶段汇总、窗口去重、复杂复用或可读性明显改善时才使用 CTE。
- 落地区：目标字段顺序必须与目标表或输出清单一致。
- 校验区：行数、主键重复、必填为空、码值越界、金额或余额方向等检查。

## 存储过程要求

- 入参、出参、异常处理、提交/回滚策略明确。
- 删除重跑、增量追加、截面刷新三种方式必须写明选择理由。
- 过程名、临时表名、中间表名应带业务含义，避免只用 `tmp1/tmp2`。
- 对大表关联要明确驱动表、过滤前置、必要索引字段和避免全表笛卡尔积的措施。

## SQL 编写规则

- 业务口径优先于技术写法；SQL 注释必须说明业务含义。
- 默认不使用 `select *`。
- 默认不使用多层 `with` / CTE。
- 目标字段顺序必须与目标表 DDL、数据表页字段清单或用户指定输出顺序一致。
- 常量字段、码值映射、条件判断、拼接字段、窗口函数和聚合字段都必须有注释说明依据。
- 金额、余额、日期、机构、客户、账号、合同号等关键字段必须明确来源表和来源字段。
- 关联条件必须写清业务键，不能只凭同名字段关联。
- 过滤条件必须能映射到业务纳入/排除规则；如果只是技术限制，要在注释中说明。
- 对日期口径要区分交易日、自然日、数据日期、会计日期、跑批日期和报送期。
- 删除重跑脚本的 `delete` 范围必须与 `insert` 范围一致。
- 汇总 SQL 必须说明分组粒度、去重逻辑、分子分母、聚合函数和空值处理。
- 明细 SQL 必须说明每条记录代表什么业务对象，以及如何保证一业务对象一行或多行的规则。
- 数据库方言不明确时，优先写 ANSI SQL 风格，并在文件头标注“数据库方言待确认”。

## 验证 SQL

生成 SQL 后必须同时给出验证 SQL 或验证清单，至少覆盖：

- 目标行数是否符合业务范围
- 目标主键或唯一性是否重复
- 必填字段是否为空
- 代码类字段是否只出现允许码值
- 金额、余额、日期字段是否存在明显异常
- 与上游来源的抽样回溯是否能对上

## 知识页同步

开发完成后不能只交付 SQL，必须同步维护相关知识页：

- 来源页：记录草案路径、依赖知识页/表结构、外部来源 wikilink、关键实现结论和待验证点。
- 报表业务口径页：如果 SQL 对应报表、接口、结果表或固定业务输出，写清整表口径、字段/指标口径、纳入排除规则、时间口径和代码类字段规则。
- 数据表页：区分目标表加工状态和源表下游使用状态。目标表字段可标记“SQL 已加工”；源表字段只能标记“被下游使用/消费”。
- 源数据表页：在“直接下游对象”记录本次 SQL 草案消费了哪些字段、生成哪个目标表/报表、证据路径和待验证状态。
- 目标数据表页：在“直接上游对象”记录本次 SQL 草案来自哪些源表/字段、处理方式、证据路径和待验证状态。
- 血缘页：至少覆盖本次 SQL 的表级链路和目标输出字段的字段级来源，并标注草案或正式证据状态。
- 概念页/系统页：如果产生新的可复用加工规则、系统边界、通用码值解释或日期口径，应更新共享知识页。
- 系统页：补齐具体对象入口、表族摘要或当前问题，不把具体对象平铺进根索引。
- `日志.md`：追加 `sql-dev` 或 `development` 记录。

## 草案与正式阶段

- SQL 尚未运行或尚未被用户确认时，相关知识页可以记录“SQL 草案待验证”，页面通常保持 `draft`。
- 血缘页可以写字段级设计血缘，但必须标注“依据 SQL 草案”，不要写成已验证生产血缘。
- 源数据表页和目标数据表页可以记录草案上下游，但确认状态必须写为“待验证”或“待确认”，不能写成生产确认消费。
- 用户确认 SQL 已运行通过、已投产或已归档为正式材料后，必须按报表 SQL ingest 工作流重新判断状态。

## 最低交付标准

- 一份位于 `工作区/SQL开发/<系统名>/` 的 SQL 或存储过程草案。
- 必要时一份同目录校验 SQL。
- 文件头列明业务目标、依赖知识页、源表、目标表、参数、运行方式和未确认点。
- 关键字段映射、过滤条件、码值转换和聚合逻辑均有依据说明。
- 已同步维护相关来源页、报表业务口径页、数据表页和血缘页；如果确无对应对象，必须在交付说明中写明原因。
- 已按血缘双向维护规则补齐：血缘页完整边、源数据表页直接下游、目标数据表页直接上游。
- 已更新对应系统页和 `日志.md`。
- 回答中列出依据页面和待确认问题。
