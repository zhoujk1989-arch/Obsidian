# 报表 SQL Ingest 工作流

当用户要求 ingest 一份新的监管报表 SQL、存储过程、视图定义、结果表加工脚本或标准接口加工 SQL 时，执行本流程。

## 工作步骤

1. 读取 `索引.md` 和对应系统页，确认系统边界和对象入口。
2. 读取 `原始材料/SQL/<系统名>/` 下的新 SQL 文件，不改写原始材料。
3. 必要时先用 `工具/血缘脚手架.py` 做第一轮结构化抽取。脚手架只作为初稿，不能替代业务解释和字段级核对。
4. 搜索相关报表业务口径页、数据表页、血缘页、来源页、系统页和概念页，优先更新已有对象。
5. 如需要监管业务来源，使用 `工具/检索外部知识库.py` 或按外部来源检索顺序检索 `my-dev-brain`，写入知识页时只记录 Obsidian wikilink。
6. 创建或更新 `知识库/来源/<系统名>/来源-...`，记录本地 SQL、外部来源 wikilink、检索关键词、关键结论和版本冲突。
7. 创建或更新 `知识库/报表业务口径/<系统名>/报表-...`，面向业务人员说明报表目的、报送范围、纳入排除规则、时间口径和字段/指标口径摘要。
8. 为涉及的核心物理表和稳定逻辑数据集创建或更新 `知识库/数据表/<系统名>/` 下的数据表页，把字段口径、码值说明、表内指标和上下游关系写回这些表页。
9. 创建或更新 `知识库/血缘/<系统名>/血缘-...`，至少覆盖目标输出字段的字段级来源。
10. 检查是否产生跨报表复用知识，例如系统边界、通用概念、共享口径、代码映射规则；如有，更新系统页或概念页。
11. 对照既有知识页检查口径冲突、旧结论替代、旧链接失效或系统边界变化。
12. 补齐来源页、报表业务口径页、数据表页、血缘页、系统页、概念页之间的摘要级互链。
13. 重新判断触达页面的 `status`。
14. 更新对应系统页中的报表入口、表族入口、来源定位、血缘入口或当前覆盖摘要；不要把具体对象平铺进根索引。
15. 向 `日志.md` 追加 ingest 记录。

## 字段与码值要求

如果目标对象是接口表、结果表或监管报送表，所有代码类字段都必须在对应数据表字段口径中直接写明：

- 码值
- 中文含义
- 映射规则
- 特殊值或常量值
- 证据来源

缺少 SQL 来源或监管来源的字段，不得写成已闭环，应进入 `Open Questions`。

## 状态判断

- 新建或大幅改写页面后必须重新判断状态。
- 只有 SQL 草案或未核验 SQL 时，相关页面通常保持 `draft`。
- 核心结论已能回溯到本地 SQL、外部来源 wikilink 或既有知识页，且互链、系统页、索引和日志已同步后，才可评估 `validated`。
- 新来源推翻或削弱既有 `validated` 结论时，相关页面必须降回 `draft` 并写明影响。

## 最低交付标准

一次报表 SQL ingest 完成的最低标准是：

- 一张来源页
- 一张报表业务口径页
- 一张血缘页
- 所有核心数据表页
- 核心字段和核心指标口径已经写入数据表页
- 接口表中的代码类字段已写入字段口径，不依赖独立码值页
- 相关系统页或通用概念页已经检查，并在需要时更新
- 相关页面的 `status` 已重新判断
- 对应系统页和 `日志.md` 已更新

如果只生成了来源页、报表业务口径页、数据表页和血缘页，却没有检查共享概念、系统页、接口表字段码值说明和旧结论冲突，视为 ingest 未完成。
