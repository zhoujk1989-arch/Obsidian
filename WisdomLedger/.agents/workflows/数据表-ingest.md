# 数据表 Ingest 工作流

当用户要求 ingest 数据表、表结构、字段字典、接口字段说明或先初始化数据表知识时，执行本流程。

## 适用输入

- DDL
- 字段字典
- 接口字段说明
- 表结构截图或表结构文件
- 用户说“先初始化数据表”“ingest 表结构”

如果同一批材料同时包含表结构和 SQL，应先执行本流程，再执行 `.agents/workflows/报表-SQL-ingest.md`。

## 工作步骤

1. 读取 `索引.md`，定位系统页；若系统页不存在，创建或更新 `知识库/概念/概念-系统-<系统名>.md`。
2. 读取 `原始材料/表结构/<系统名>/` 下的 DDL、字段字典、接口说明或其他本地表结构材料，不改写原始材料。
3. 搜索同系统下已有数据表页、来源页、血缘页、报表业务口径页、系统页和概念页，优先更新已有对象，不默认新建近似重复页。
4. 判断数据表所属系统、表类型和知识边界：
   - 物理表
   - 稳定逻辑数据集
   - 接口表
   - 结果表
   - 监管报送表
5. 创建或更新 `知识库/来源/<系统名>/来源-...`，记录本地 DDL、字段字典、接口说明、已核验的外部 `my-dev-brain` wikilink、检索关键词和关键发现。
6. 创建或更新 `知识库/数据表/<系统名>/数据表-...`，至少写清：
   - 表在监管链路中的角色
   - 主键与核心标识
   - 字段清单与字段口径
   - 字段落地状态
   - 表内指标口径（如有）
   - 上游对象、下游对象和相关报表
   - 相关来源页和血缘页
7. 检查是否产生可复用知识，例如系统边界、通用表角色、共享字段规则；如有，更新系统页或通用概念页。
8. 对照既有页面检查字段定义冲突、码值冲突、表归属变化或旧结论被削弱。
9. 重新判断触达页面的 `status`。
10. 更新对应系统页中的表族入口、来源定位或当前覆盖摘要；不要把具体表页平铺进根索引。
11. 向 `日志.md` 追加数据表 ingest 记录。

## 字段落地状态

字段落地状态必须区分：

- 本地表结构字段：DDL 或字段字典中已经存在。
- SQL 已加工字段：生成或写入本数据表的 SQL、过程或视图中已经明确赋值、映射或聚合。
- 当前缺口字段：表结构已存在，但 SQL 来源、填报规则或是否报送尚未闭环。
- 不适用字段：有明确依据说明当前场景不需要报送或不参与加工。
- 监管要求字段：仅当外部监管来源或既有报表业务口径页已经确认需要报送时才标注。

下游 SQL 读取本表字段，不等于本表字段已加工落地。源表字段被下游使用时，应写成“字段被消费情况”或“下游使用情况”。

## 代码类字段

如果该表是接口表、结果表或监管报送表，所有代码类字段必须在字段口径中直接写清：

- 码值
- 中文含义
- 映射规则
- 特殊值或常量值
- 证据来源

不得只写“见码表”，不得为了复用强行新建独立码值页。

## 血缘边界

- 只有表结构材料、尚无 SQL 加工证据时，不要硬造血缘。
- 如果已有相关血缘页，补齐数据表页到血缘页的回链。
- 如果已有 SQL 或血缘页证明本表是上游源表，必须在本表页“直接下游对象”记录被消费对象、消费字段摘要和证据。
- 如果已有 SQL 或血缘页证明本表是下游目标表，必须在本表页“直接上游对象”记录来源对象、来源字段摘要和证据。
- 如果尚无血缘页，只有在已有明确上下游证据时才创建或更新血缘页；否则写成普通文本“待补（血缘页名）”，不要制造不存在 wikilink。
- 上下游摘要的双向维护细则见 `.agents/rules/血缘双向维护规则.md`。

## 最低交付标准

一次数据表 ingest 完成的最低标准是：

- 一张来源页
- 一张数据表页
- 字段清单和字段落地状态
- 必要的 `Open Questions`
- 对应系统页更新
- `日志.md` 记录

如触达接口表、结果表或监管报送表，还必须覆盖代码类字段码值说明。
