# WisdomLedger 智能体规范

这个仓库是一个由 LLM 持续维护的知识库，用来把监管报表 SQL 沉淀为：

1. 业务可读的自然语言口径
2. 可持续维护的血缘文档
3. 可复用的报表与数据表知识

整体运行方式遵循 Karpathy 提出的 `LLM Wiki` 三层模式：

- `原始材料/`：不可变的原始材料层。本仓库只存 SQL 和本地技术材料。
- `知识库/`：由 LLM 维护的知识层。报表页、数据表页、血缘页是主载体。
- `AGENTS.md`：约束 LLM 工作方式的规范文件，用来保证结构、流程和维护纪律。

监管原文、填报说明、制度文件和指标来源材料不保存在本仓库，而是关联外部 Obsidian 知识库：

- `/Users/zhoujingkun/Documents/GitHub/Obsidian/my-dev-brain`

## 仓库结构

- `原始材料/SQL/`：按系统分目录存放报表 SQL、中间加工 SQL、视图定义等本地技术材料。
- `原始材料/表结构/`：按系统分目录存放表结构、字段字典、接口字段说明等本地技术材料。
- `知识库/来源/`：按系统分目录；每个来源包一页，记录本地 SQL 与外部监管来源的关联。
- `知识库/报表/`：按系统分目录；每张监管报表一页。
- `知识库/数据表/`：按系统分目录；每张物理表或稳定逻辑数据集一页，默认同时承载字段和表内指标口径。
- `知识库/血缘/`：按系统分目录；端到端血缘页。
- `知识库/概念/`：可复用的概念、术语和方法页。
- `模板/`：页面模板。用于复制结构，不保留占位内容。
- `索引.md`：内容导向的总目录。
- `日志.md`：按时间追加的操作记录。

## 路径分层规则

- 系统是路径层级，不只是页面正文属性。
- 除 `知识库/概念/` 外，所有知识页和 SQL 文件默认都应放在对应系统目录下。
- 不同系统的对象，即使同名，也不能放在同一路径下混用。
- 跨系统内容不建“混合目录”，而是放在主系统目录下，并在正文中显式写明涉及的其他系统。

推荐路径形态：

- `原始材料/SQL/<系统名>/...`
- `原始材料/表结构/<系统名>/...`
- `知识库/来源/<系统名>/来源-...`
- `知识库/报表/<系统名>/报表-...`
- `知识库/数据表/<系统名>/数据表-...`
- `知识库/血缘/<系统名>/血缘-...`

例如：

- `原始材料/SQL/核心系统/对公存款余额日报.sql`
- `知识库/数据表/核心系统/数据表-客户主表.md`
- `知识库/血缘/监管报送平台/血缘-对公存款余额日报.md`


## 命名规则

- 来源页：`来源-...`
- 报表页：`报表-...`
- 数据表页：`数据表-...`
- 血缘页：`血缘-...`
- 概念页：`概念-...`

页面标题优先使用中文业务名称。技术标识放在 frontmatter 或专门章节中。

目录路径优先使用系统名进行分层，页面标题不强制带系统名；只有在同名对象确实需要区分时，才在页面标题中追加系统名。

## 正式知识对象

- 单个报表 ingest 不只产出“报表包”。
- 以下对象都属于正式知识对象，agent 在 ingest、query、lint 时都必须检查是否需要更新：
  - 来源页
  - 报表页
  - 数据表页
  - 血缘页
  - 系统页与通用概念页
- 如果新信息具有跨报表复用价值，优先更新共享知识对象，而不是只写进当前报表页。

## 核心原则

1. 原始材料不可变，不要改写 `原始材料/` 下的文件。
2. 先表达业务含义，再描述 SQL 实现。
3. 任何重要结论都应能追溯到证据：
   - 本地 SQL 文件
   - `my-dev-brain` 中的外部监管来源
   - 已存在的知识库页面
4. 优先更新已有页面，不要制造近似重复页。
5. 如果 SQL 语义有歧义，必须在 `## Open Questions` 中显式写出。
6. 当用户提出了可复用的问题和结论，应把它们回写到知识库，而不是只留在对话里。
7. 不要把大段监管原文复制到本仓库；这里只记录路径、检索关键词和提炼后的结论。
8. 所有 SQL、报表、指标、数据表、字段、来源、血缘页面都必须通过目录路径体现所属系统。
9. 系统间默认彼此独立；跨系统关系通过血缘页或映射页表达，不通过共享路径表达。
10. 字段口径和指标口径统一写入数据表页，不再维护独立字段页或指标页。
11. 代码类字段的码值含义、取值范围和映射规则，默认直接写入接口表或结果表对应字段口径，不再单独维护码值页。
12. 如果同一套码值被多个接口表复用，也优先分别写入各自字段口径；不要为“复用”而强行抽成独立页面。
13. ingest、query、lint 都必须优先更新已有页面和共享知识页，而不是不断新增孤立页面。
14. 新来源如果挑战旧结论，必须显式标注冲突、更新相关页面，并在必要时降级页面状态；不能静默覆盖。
15. 不允许在知识页中制造指向不存在页面的占位 wikilink；待补内容一律写成普通文本“待补充”或“待补（目标页名）”。

## 必备 Frontmatter

每个知识库页面都应包含：

```yaml
---
type: report|table|lineage|source|concept
id: 报表-...
status: draft|validated
updated: YYYY-MM-DD
tags:
  - regulatory
  - lineage
---
```

来源页建议额外包含：

```yaml
external_vault: ../my-dev-brain
external_paths: []
search_keywords: []
```

除非确实稳定且有用，不要随意增加额外字段。

## 页面状态规则

- `draft`：
  - 新建页面默认使用 `draft`
  - 仍有关键证据缺口
  - 必填章节未完成
  - 仍存在阻塞型 `Open Questions`
  - 仍存在未修复死链、明显占位内容或互链不完整
- `validated`：
  - 必填章节已完成
  - 核心结论已能回溯到本地 SQL、外部路径或既有知识页
  - 互链、索引与日志已同步
  - 不再存在阻塞型缺口
  - 即使仍保留 `Open Questions`，也只能是非阻塞问题
- 状态变更规则：
  - 新建或大幅改写页面后，agent 必须在本次工作末尾重新判断状态
  - 如果新来源推翻或削弱既有 `validated` 结论，相关页面必须降回 `draft`
  - 不能把 `validated` 当作“写完了”的装饰标签；它代表当前版本已经完成最小验证闭环

## Ingest 工作流

当用户要求 ingest 一份新的监管报表 SQL 时：

1. 读取 `原始材料/SQL/<系统名>/` 下的新 SQL 文件；必要时先用 `工具/血缘脚手架.py` 做第一轮结构抽取。
2. 在本库中先搜索相关报表页、数据表页、血缘页、系统页和概念页，优先更新已有对象，不要默认新建。
3. 用 `工具/检索外部知识库.py` 或按既定顺序检索 `/Users/zhoujingkun/Documents/GitHub/Obsidian/my-dev-brain`，定位报表页、主题页和原文页。
4. 创建或更新一页 `知识库/来源/<系统名>/来源-...`，记录本地文件、外部路径、检索关键词和关键结论，不复制原文。
5. 创建或更新对应的 `知识库/报表/<系统名>/报表-...` 页面。
6. 为涉及的核心物理表和稳定逻辑数据集创建或更新 `知识库/数据表/<系统名>/` 下的数据表页，并把字段口径、指标口径写回这些表页。
7. 如果目标对象是接口表、结果表或监管报送表，所有代码类字段都必须在对应字段口径中直接写明：
   - 码值
   - 中文含义
   - 映射规则
   - 特殊值或常量值
   - 证据来源
8. 创建或更新 `知识库/血缘/<系统名>/血缘-...` 页面，至少覆盖目标输出字段的字段级来源。
9. 检查这次 ingest 是否产生了跨报表复用知识：
   - 新的系统边界说明
   - 新的通用概念
   如果有，必须同步更新对应共享知识页，而不是只留在当前报表包里。
10. 对照既有知识页检查是否出现口径冲突、旧结论被替代、旧链接失效或系统边界变化：
   - 如果冲突可确认，直接更新旧页并写明变更影响
   - 如果冲突尚不能确认，在相关页 `Open Questions` 中显式记录
   - 如果既有 `validated` 页面被新来源削弱，降为 `draft`
11. 补齐报表页、数据表页、血缘页、来源页、系统页、概念页之间的双向互链。
12. 重新判断这次触达页面的 `status`。
13. 更新 `索引.md`。
14. 向 `日志.md` 追加一条时间记录，记录本次 ingest 触达了哪些关键对象，以及是否发现冲突或补了共享知识页。

说明：

- 一次 ingest 可以也应该同时更新多页；不要把它理解成“一个 SQL 只配一个报表包”。
- 如果一份新来源只生成了来源页、报表页、数据表页和血缘页，却没有检查共享概念、系统页、接口表字段码值说明和旧结论冲突，默认视为 ingest 未完成。

## 如何把 SQL 翻译成业务语言

对于每一份 SQL，先抽取：

- 报表目的
- 统计粒度
- 维度
- 度量
- 纳入规则
- 排除规则
- 时间逻辑
- 来源数据表
- 关联逻辑
- 转换阶段
- 最终输出字段

然后把它改写成业务语言，重点回答：

- “到底在统计什么？”
- “统计发生在什么日期或粒度上？”
- “哪些对象被纳入，哪些被排除？”
- “哪些来源表是权威来源？”
- “哪些过滤条件会实质性改变业务口径？”

不要把 SQL 直接原样搬进正文。先用业务语言解释含义，再把 SQL 表达式作为证据保留。

对于监管原文：

- 不要在这里复制全文
- 只保留简洁的业务结论
- 记录 `my-dev-brain` 中的精确路径
- 记录可重复使用的检索关键词

## 报表页写法规则

报表页默认面向业务人员阅读，不要把报表页写成 SQL 注释翻译稿或开发说明。

### 明细类报表

如果目标报表是明细类报表，必须完整覆盖两部分：

1. 整表报送范围
2. 每个报送字段的字段级口径

最低要求：

- 必须说明整张表到底报送哪些业务对象
- 必须说明哪些对象纳入、哪些对象排除
- 必须说明这是全量报送、增量报送还是截面报送
- 必须逐字段说明口径，不能只写重点字段
- 每个字段都要说明：
  - 字段含义
  - 取值或映射逻辑
  - 来源说明
  - 特殊规则

明细类字段口径中的“来源说明”应优先使用自然中文句式，例如：

- `来源于监管集市机构表的机构地址字段`
- `来源于客户主表的客户编号字段`

不要在该列直接裸写技术表名和字段名，如：

- 不推荐：`ORG_ADD / L_PUBL_ORG_BRA`

如果确实需要保留技术标识，应放在“规则说明”中作为证据补充，而不是替代业务表达。

### 汇总类报表

如果目标报表是汇总类报表，必须完整覆盖每个指标的取数规则，不能只写核心指标。

最低要求：

- 必须逐指标说明，不可省略
- 每个指标都要说明：
  - 指标名称
  - 统计对象
  - 取数规则
  - 过滤条件
  - 聚合方式
  - 依赖来源
- 如果是派生指标，必须说明依赖哪些基础指标
- 如果存在分子、分母、去重口径、时点口径，必须写清楚

### 通用要求

- 业务解释优先，SQL 细节后置
- 不允许报表页只停留在“源表 + where 条件 + insert 目标表”的摘要层
- 代码类字段必须写清楚码值或映射来源
- 常量字段、拼接字段、判断字段、窗口函数字段都必须显式说明
- 如果页面中出现技术字段名，应同时给出业务名称或中文含义
- 模板以 `模板/报表模板.md` 为准，新增报表页默认按该模板落地

## 数据表页写法规则

数据表页是字段口径和表内指标口径的默认承载页。

最低要求：

- 必须说明该表在监管链路中的角色
- 必须覆盖：
  - 主键与核心标识
  - 关键字段与口径
  - 表内指标口径（如有）
  - 上下游对象
- 字段和表内指标统一在数据表页内维护，不再拆成独立页面

如果该表是接口表、结果表或监管报送表，最低要求再增加：

- 代码类字段必须逐字段写清码值含义
- 不能只写“见码表”或“参考字典”
- 必须把常量值、枚举值、压缩映射和特殊取值直接写到字段口径中
- 如果某个字段的码值很多，可在该字段下补充子表，但仍需留在本表页内

## 血缘规则

每张血缘页都应包含：

1. 一段业务链路摘要
2. 节点列表
3. 表级边列表
4. 字段级边列表
5. 条件允许时提供 Mermaid 图
6. 已知缺口、人工假设或未确认点

血缘页默认面向技术人员使用，主要服务于：

- 影响分析
- 反向溯源
- 字段口径核对
- 变更评估

因此血缘页不能只停留在“表到表”或“CTE 到 CTE”的粗粒度摘要层。

并区分三类血缘：

- 物理血缘：表到表
- 逻辑血缘：CTE、中间结果、派生指标
- 语义血缘：监管指标 -> 业务指标 -> SQL 表达式 -> 来源字段

### 表级血缘要求

- 说明源表、目标表、中间层、过程或脚本之间的关系
- 说明每条边对应的主要处理动作，例如过滤、关联、聚合、插入、删除重算
- 表级图用于快速看全链路，不替代字段级血缘

### 字段级血缘要求

如果血缘页对应监管报表、标准接口表、核心下游结果表，默认应补充字段级血缘。

最低要求：

- 至少覆盖所有目标输出字段
- 每个目标字段都要说明来源字段、来源对象、处理逻辑
- 如果是直接映射，要明确写“直接映射”
- 如果是拼接、条件判断、码值映射、聚合、窗口函数、常量赋值，也要显式写出
- 如果多个目标字段来自同一源字段，不能省略任何一个目标字段
- 如果一个目标字段依赖多个源字段，要全部列出

字段级边列表建议至少包含：

- 源对象
- 源字段
- 目标对象
- 目标字段
- 处理逻辑
- 关系类型
- 证据

其中关系类型建议使用：

- 直接映射
- 条件映射
- 拼接派生
- 聚合派生
- 窗口派生
- 常量赋值
- 码值转换

字段级血缘默认面向技术人员，因此字段级边列表和字段级 Graph 中：

- 源字段、目标字段优先使用英文字段名或技术字段名
- 不要把中文字段名作为字段级血缘的主展示名称
- 如有必要，可在“处理逻辑”或补充说明中附带中文含义
- 推荐写法：
  - 源对象：中文或稳定技术对象名均可
  - 源字段：`ORG_ADD`
  - 目标字段：`A010016`

### Graph 画法要求

- 每张血缘页建议至少提供两层图：
  - 总览图：展示表级和过程级主链路
  - 详细图：展示字段级关联关系
- 字段级图应尽量体现“原表.原字段 -> 目标表.目标字段”
- 如果字段过多，可以分主题拆成多张 Mermaid 图，但不能省略目标字段
- 如果同一字段存在多来源回溯，应在图或边表中写清“本级/上级/回溯补齐”等关系
- 图是为技术分析服务，优先完整和可追溯，不必追求简洁美观

### 血缘互链规则

为了支持影响分析和反向溯源，血缘关系不能只存在于血缘页本身，必须形成对象页与血缘页的双向回链。

最低要求：

- 每张血缘页都要显式列出：
  - 直接上游对象
  - 直接下游对象
  - 相关数据表页
  - 相关报表页
- 每个被血缘页引用的核心数据表页，都应回链到对应血缘页
- 每个被血缘页引用的核心报表页，都应回链到对应血缘页
推荐理解方式：

- 从血缘页出发，能跳到上游对象和下游对象
- 从数据表页出发，能看到“我被哪些血缘页使用”
- 从报表页出发，能看到“这张报表对应哪条血缘链路”
### 模板联动要求

在创建或更新以下页面时，应同步检查互链是否完整：

- `模板/血缘模板.md`
- `模板/数据表模板.md`
- `模板/报表模板.md`
- `模板/来源模板.md`

如果新增了一张血缘页，至少要同步检查：

1. 上游数据表页是否已回链该血缘页
2. 下游数据表页是否已回链该血缘页
3. 报表页是否已回链该血缘页
4. 上下游表页和报表页是否已补齐回链

## 查询工作流

当回答问题时：

1. 先读 `索引.md`
2. 打开最相关的知识库页面
3. 如果问题依赖监管原文含义，再去检索 `my-dev-brain`
4. 结合页面和外部来源做综合回答
5. 回答中必须显式给出依据
6. 如果答案具有长期价值，把它回写到知识库

### Query 引用规则

- 回答中的关键结论不能裸写，至少要带一条本库引用。
- 本库引用优先使用页面名，例如：
  - `依据：[[报表-...]]`
  - `依据：[[数据表-...]]、[[血缘-...]]`
- 如果结论直接依赖外部监管原文或 `my-dev-brain` 的页面含义，必须同时给出外部路径或 `external_paths` 中的命中来源。
- 如果回答综合了多个页面，应在回答结尾集中列出“引用页面”或“依据页面”。
- 如果当前知识库里还没有合适承载页，但这个问答具有长期复用价值，应创建或更新下列之一：
  - 相关报表页
  - 相关数据表页
  - 相关血缘页
  - 通用概念页
  - 比较页

### Query 回写规则

- 满足以下任一条件，就不应只停留在对话里：
  - 用户问到了可复用的口径差异
  - 用户澄清了一个长期存在的歧义
  - 回答形成了跨报表可复用的结论
  - 回答发现了既有页面的错误、过时或矛盾
- 回写后要同步更新相关页面的 `updated`、`status`、`索引.md` 和必要的 `日志.md`。

## 巡检工作流

lint 不是泛泛“看一眼”，而是一种可执行维护动作。默认输入为：

- `索引.md`
- `日志.md`
- 最近修改页
- 相关来源页、报表页、数据表页、血缘页、概念页

默认检查项包括：

- 孤儿页面或几乎没有入链的页面
- 缺少上下游链接的数据表页
- 在血缘页中提到、但还没有沉淀到表页或概念页的关键对象或规则
- 业务定义与 SQL 过滤条件是否矛盾
- 是否存在被新 SQL 或新规则替代的陈旧页面
- 来源页是否缺少 `external_paths`
- 是否存在指向不存在页面的 wikilink
- 是否仍有“待补充”出现在必填章节
- `validated` 页面是否仍满足验证条件
- 接口表中的代码类字段是否已写清码值说明

### Lint 处理规则

- 对于可直接修复的问题，agent 应直接修复，而不是只列问题清单。
- 对于暂时无法修复的问题，必须：
  - 写入相关页面的 `Open Questions`
  - 在必要时降级页面状态
  - 在 `日志.md` 中记录一次 `audit` 或 `lint` 条目
- Lint 如果修了互链、状态、索引或共享知识页，也要同步更新这些对象。
- 死链规则：
  - 知识页中的不存在 wikilink 视为错误
  - `日志.md`、`AGENTS.md`、`模板/` 中作为历史记录或示例出现的占位链接，不纳入知识图完整性判断
  - 如果对象尚未建立，使用普通文本 `待补充` 或 `待补（目标页名）`，不要制造空链接

### Lint 输出要求

- 如果本次 lint 没有改文件，也要给出明确结论：检查范围、未发现问题或仍存风险
- 如果本次 lint 改了文件，至少要更新：
  - 被修复页面
  - 必要的 `索引.md`
  - 必要的 `日志.md`
  - 被影响页面的 `status`

## 写作风格

- 业务解释优先使用简洁中文
- 英文或 `snake_case` 只用于技术标识
- 优先使用要点和表格，不写长篇大段说明
- Obsidian 互链使用 `[[报表-对公存款余额日报]]` 这种格式
- 代码片段只保留证明结论所必需的最短部分
- 对外部来源，只记录 `my-dev-brain` 的路径或相对路径，不复制正文

## 每张新报表的最低交付标准

一份报表 ingest 完成的最低标准是：

- 一张带 `external_paths` 的来源页
- 一张报表页
- 一张血缘页
- 所有核心数据表页
- 核心字段和核心指标口径已经写入数据表页
- 接口表中的代码类字段已写入字段口径，不依赖独立码值页
- 相关共享知识页已经检查，并在需要时更新：
  - 系统页
  - 通用概念页
- 相关页面的 `status` 已重新判断
- 已更新的 `索引.md` 和 `日志.md`

独立指标页或字段页不再维护。
