# WisdomLedger 智能体规范

这个仓库是一个由 LLM 持续维护的知识库，用来把监管报表 SQL 沉淀为：

1. 业务可读的自然语言口径
2. 可持续维护的血缘文档
3. 可复用的报表、指标、数据表和字段知识

整体运行方式遵循 Karpathy 提出的 `LLM Wiki` 三层模式：

- `原始材料/`：不可变的原始材料层。本仓库只存 SQL 和本地技术材料。
- `知识库/`：由 LLM 维护的知识层。报表页、指标页、字段页、血缘页都在这里。
- `AGENTS.md`：约束 LLM 工作方式的规范文件，用来保证结构、流程和维护纪律。

监管原文、填报说明、制度文件和指标来源材料不保存在本仓库，而是关联外部 Obsidian 知识库：

- `/Users/zhoujingkun/Documents/GitHub/Obsidian/my-dev-brain`

## 仓库结构

- `原始材料/SQL/`：按系统分目录存放报表 SQL、中间加工 SQL、视图定义等本地技术材料。
- `知识库/来源/`：按系统分目录；每个来源包一页，记录本地 SQL 与外部监管来源的关联。
- `知识库/报表/`：按系统分目录；每张监管报表一页。
- `知识库/指标/`：按系统分目录；每个业务指标或监管指标一页。
- `知识库/数据表/`：按系统分目录；每张物理表或稳定逻辑数据集一页。
- `知识库/字段/`：按系统分目录；每个关键字段一页。
- `知识库/血缘/`：按系统分目录；端到端血缘页。
- `知识库/概念/`：可复用的概念、术语和方法页。
- `模板/`：页面模板。用于复制结构，不保留占位内容。
- `索引.md`：内容导向的总目录。
- `日志.md`：按时间追加的操作记录。

## 路径分层规则

- 系统是路径层级，不只是页面正文属性。
- 除 `知识库/概念/` 外，所有知识页和 SQL 文件默认都应放在对应系统目录下。
- 不同系统的对象，即使同名，也不能放在同一路径下混用。
- 跨系统内容不建“混合目录”，而是放在主系统目录下，并在正文中显式写明涉及的其他系统。

推荐路径形态：

- `原始材料/SQL/<系统名>/...`
- `知识库/来源/<系统名>/来源-...`
- `知识库/报表/<系统名>/报表-...`
- `知识库/指标/<系统名>/指标-...`
- `知识库/数据表/<系统名>/数据表-...`
- `知识库/字段/<系统名>/字段-...`
- `知识库/血缘/<系统名>/血缘-...`

例如：

- `原始材料/SQL/核心系统/对公存款余额日报.sql`
- `知识库/数据表/核心系统/数据表-客户主表.md`
- `知识库/字段/CRM/字段-客户类型代码.md`
- `知识库/血缘/监管报送平台/血缘-对公存款余额日报.md`

## 命名规则

- 来源页：`来源-...`
- 报表页：`报表-...`
- 指标页：`指标-...`
- 数据表页：`数据表-...`
- 字段页：`字段-...`
- 血缘页：`血缘-...`
- 概念页：`概念-...`

页面标题优先使用中文业务名称。技术标识放在 frontmatter 或专门章节中。

目录路径优先使用系统名进行分层，页面标题不强制带系统名；只有在同名对象确实需要区分时，才在页面标题中追加系统名。

## 核心原则

1. 原始材料不可变，不要改写 `原始材料/` 下的文件。
2. 先表达业务含义，再描述 SQL 实现。
3. 任何重要结论都应能追溯到证据：
   - 本地 SQL 文件
   - `my-dev-brain` 中的外部监管来源
   - 已存在的知识库页面
4. 优先更新已有页面，不要制造近似重复页。
5. 如果 SQL 语义有歧义，必须在 `## Open Questions` 中显式写出。
6. 当用户提出了可复用的问题和结论，应把它们回写到知识库，而不是只留在对话里。
7. 不要把大段监管原文复制到本仓库；这里只记录路径、检索关键词和提炼后的结论。
8. 所有 SQL、报表、指标、数据表、字段、来源、血缘页面都必须通过目录路径体现所属系统。
9. 系统间默认彼此独立；跨系统关系通过血缘页或映射页表达，不通过共享路径表达。

## 必备 Frontmatter

每个知识库页面都应包含：

```yaml
---
type: report|metric|table|field|lineage|source|concept
id: 报表-...
status: draft|validated
updated: YYYY-MM-DD
tags:
  - regulatory
  - lineage
---
```

来源页建议额外包含：

```yaml
external_vault: ../my-dev-brain
external_paths: []
search_keywords: []
```

除非确实稳定且有用，不要随意增加额外字段。

## Ingest 工作流

当用户要求 ingest 一份新的监管报表 SQL 时：

1. 读取 `原始材料/SQL/<系统名>/` 下的新 SQL 文件。
2. 到 `/Users/zhoujingkun/Documents/GitHub/Obsidian/my-dev-brain` 检索对应的报表页、主题页和原文页。
3. 创建或更新一页 `知识库/来源/<系统名>/来源-...`，只记录外部路径和检索关键词，不复制原文。
4. 创建或更新对应的 `知识库/报表/<系统名>/报表-...` 页面。
5. 为 SQL 暴露出的核心业务指标创建或更新 `知识库/指标/<系统名>/` 下的指标页。
6. 为涉及的核心物理表和逻辑数据集创建或更新 `知识库/数据表/<系统名>/` 下的数据表页。
7. 为关键过滤、关联、聚合字段创建或更新 `知识库/字段/<系统名>/` 下的字段页。
8. 创建或更新 `知识库/血缘/<系统名>/血缘-...` 页面。
9. 更新 `索引.md`。
10. 向 `日志.md` 追加一条时间记录。

## 外部来源检索顺序

当需要在 `my-dev-brain` 中查找监管报表或指标来源时，按以下顺序检索：

1. `04-综合/监管系统-报表清单.md`
2. `03-实体/`
3. `02-主题/`
4. `01-资料库/`
5. `05-日志/`，仅在需要追踪历史 ingest 上下文时使用

检索关键词优先使用：报表编码、报表全名、指标名称、监管系统名称。

## 如何把 SQL 翻译成业务语言

对于每一份 SQL，先抽取：

- 报表目的
- 统计粒度
- 维度
- 度量
- 纳入规则
- 排除规则
- 时间逻辑
- 来源数据表
- 关联逻辑
- 转换阶段
- 最终输出字段

然后把它改写成业务语言，重点回答：

- “到底在统计什么？”
- “统计发生在什么日期或粒度上？”
- “哪些对象被纳入，哪些被排除？”
- “哪些来源表是权威来源？”
- “哪些过滤条件会实质性改变业务口径？”

不要把 SQL 直接原样搬进正文。先用业务语言解释含义，再把 SQL 表达式作为证据保留。

对于监管原文：

- 不要在这里复制全文
- 只保留简洁的业务结论
- 记录 `my-dev-brain` 中的精确路径
- 记录可重复使用的检索关键词

## 血缘规则

每张血缘页都应包含：

1. 一段业务链路摘要
2. 节点列表
3. 边列表表格
4. 条件允许时提供 Mermaid 图
5. 已知缺口、人工假设或未确认点

并区分三类血缘：

- 物理血缘：表到表
- 逻辑血缘：CTE、中间结果、派生指标
- 语义血缘：监管指标 -> 业务指标 -> SQL 表达式 -> 来源字段

## 查询工作流

当回答问题时：

1. 先读 `索引.md`
2. 打开最相关的知识库页面
3. 如果问题依赖监管原文含义，再去检索 `my-dev-brain`
4. 结合页面和外部来源做综合回答
5. 如果答案具有长期价值，把它回写到知识库

## 巡检工作流

定期检查：

- 孤儿页面或几乎没有入链的页面
- 缺少证据段的指标页
- 缺少上下游链接的数据表页
- 在血缘页中提到、但还没有独立页面的节点
- 业务定义与 SQL 过滤条件是否矛盾
- 是否存在被新 SQL 或新规则替代的陈旧页面
- 来源页是否缺少 `external_paths`

## 写作风格

- 业务解释优先使用简洁中文
- 英文或 `snake_case` 只用于技术标识
- 优先使用要点和表格，不写长篇大段说明
- Obsidian 互链使用 `[[报表-对公存款余额日报]]` 这种格式
- 代码片段只保留证明结论所必需的最短部分
- 对外部来源，只记录 `my-dev-brain` 的路径或相对路径，不复制正文

## 每张新报表的最低交付标准

一份报表 ingest 完成的最低标准是：

- 一张带 `external_paths` 的来源页
- 一张报表页
- 一张血缘页
- 所有核心指标页
- 所有核心数据表页
- 已更新的 `索引.md` 和 `日志.md`
