# WisdomLedger 智能体总章程

本仓库是一个由 LLM 持续维护的监管报表知识库，用来把监管报表 SQL、表结构和本地技术材料沉淀为：

1. 业务可读的自然语言口径
2. 可持续维护的字段与表级血缘
3. 可复用的报表、数据表、系统和概念知识

整体运行遵循 `LLM Wiki` 三层模式：

- `原始材料/`：不可变原始材料层。本仓库只存 SQL、DDL、字段字典和本地技术材料。
- `知识库/`：由 LLM 维护的知识层。来源页、报表业务口径页、数据表页、血缘页和概念页是主载体。
- `AGENTS.md` 与 `.agents/`：智能体工作规范层。`AGENTS.md` 只保留总规则和任务路由，细则按主题拆入 `.agents/workflows/` 和 `.agents/rules/`。

监管原文、填报说明、制度文件和指标来源材料不保存在本仓库，而是关联外部 Obsidian 知识库：

- 外部 vault：`my-dev-brain`
- 外部来源依赖只记录 Obsidian wikilink，例如 `[[03-实体/一表通-1.1机构信息]]`。
- 不记录外部 vault 的本机绝对路径，不复制大段监管原文。

## 任务路由

开始任何工作前，先按用户意图读取对应专题规范。`AGENTS.md` 中的仓库级规则始终优先；专题文件负责执行细则。

| 用户意图或输入材料                                 | 默认流程          | 必读规范                                                        |
| ----------------------------------------- | ------------- | ----------------------------------------------------------- |
| DDL、字段字典、接口字段说明、表结构截图、“先初始化数据表”           | 数据表 ingest    | `.agents/workflows/数据表-ingest.md`                           |
| 报表 SQL、存储过程、视图定义、结果表加工脚本、“ingest 报表 SQL”  | 报表 SQL ingest | `.agents/workflows/报表-SQL-ingest.md`                        |
| 写 SQL、补 SQL、生成 SQL、写存储过程、生成视图、加工脚本、校验 SQL | SQL/存储过程开发    | `.agents/workflows/SQL开发.md`；同时使用 `.agents/skills/sql-dev/` |
| 只问口径、差异、字段含义、历史结论                         | 查询与必要回写       | `.agents/workflows/查询与回写.md`                                |
| 改表、改字段、字段口径变化、血缘影响分析、下游影响分析               | 血缘影响分析        | `.agents/rules/血缘影响分析规则.md`                                 |
| 巡检、检查、lint、死链、一致性、状态核对                    | 巡检/lint       | `.agents/workflows/巡检-lint.md`                              |
| 页面写作、报表页/数据表页/血缘页结构                       | 页面写作规则        | `.agents/rules/页面写作规则.md`                                   |
| 外部监管来源、`my-dev-brain`、wikilink            | 外部来源规则        | `.agents/rules/外部来源规则.md`                                   |
| 索引、系统入口、日志                                | 索引与日志规则       | `.agents/rules/索引与日志规则.md`                                  |

如果同一批材料同时包含表结构和 SQL，应先执行数据表初始化或更新，再执行报表 SQL ingest，并在 `日志.md` 中写明两阶段都已处理。

## 仓库结构

- `原始材料/SQL/<系统名>/`：按系统存放报表 SQL、中间加工 SQL、视图定义等本地技术材料。
- `原始材料/表结构/<系统名>/`：按系统存放 DDL、字段字典、接口字段说明等本地技术材料。
- `知识库/来源/<系统名>/`：来源证据包页面。
- `知识库/报表业务口径/<系统名>/`：报表业务口径页面。
- `知识库/数据表/<系统名>/`：物理表或稳定逻辑数据集页面，默认承载字段和表内指标口径。
- `知识库/血缘/<系统名>/`：端到端血缘页面。
- `知识库/概念/`：通用概念、系统页和方法页。系统页命名为 `概念-系统-<系统名>.md`。
- `模板/`：页面和 SQL 模板。
- `工具/`：外部知识库检索、SQL 血缘脚手架等工具。
- `工作区/SQL开发/<系统名>/`：agent 生成的 SQL 草案、存储过程草案、校验脚本和说明。草案确认前不放入 `原始材料/`。
- `索引.md`：总索引，只保留系统摘要、模块说明和任务路由；具体对象入口下沉到系统页。
- `日志.md`：按时间追加操作记录。

## 路径分层规则

- 系统是路径层级，不只是页面正文属性。
- 除 `知识库/概念/` 外，所有知识页和 SQL 文件默认都应放在对应系统目录下。
- 不同系统的对象，即使同名，也不能放在同一路径下混用。
- 跨系统内容不建“混合目录”，而是放在主系统目录下，并在正文中显式写明涉及的其他系统。

推荐路径形态：

- `原始材料/SQL/<系统名>/...`
- `原始材料/表结构/<系统名>/...`
- `知识库/来源/<系统名>/来源-...`
- `知识库/报表业务口径/<系统名>/报表-...`
- `知识库/数据表/<系统名>/数据表-...`
- `知识库/血缘/<系统名>/血缘-...`
- `工作区/SQL开发/<系统名>/...`

## 命名规则

- 来源页：`来源-...`
- 报表业务口径页：`报表-...`
- 数据表页：`数据表-...`
- 血缘页：`血缘-...`
- 概念页：`概念-...`

页面标题优先使用中文业务名称。技术标识放在 frontmatter 或专门章节中。目录路径优先使用系统名进行分层，页面标题不强制带系统名；只有在同名对象确实需要区分时，才在页面标题中追加系统名。

## 正式知识对象

单个报表 ingest 不只产出“报表包”。以下对象都属于正式知识对象，agent 在 ingest、query、lint、SQL 开发时都必须检查是否需要更新：

- 来源页
- 报表业务口径页
- 数据表页
- 血缘页
- 系统页与通用概念页
- `索引.md` 和 `日志.md`

如果新信息具有跨报表复用价值，优先更新共享知识对象，而不是只写进当前报表业务口径页。

## 知识对象边界

| 对象 | 主职责 | 应避免 |
| --- | --- | --- |
| 来源页 | 记录证据包：本地 SQL/DDL/表结构材料、外部 `my-dev-brain` wikilink、检索关键词、关键发现、版本冲突 | 不维护完整字段口径，不复制报表字段表，不展开字段级血缘 |
| 报表业务口径页 | 业务视角说明：报表报什么、报谁、报送范围、纳入排除规则、字段或指标是否属于监管报送要求 | 不作为字段技术来源的唯一承载页，不复制数据表页完整字段字典，不展开完整字段级血缘 |
| 数据表页 | 字段与表内指标主档：主键、字段含义、码值、取值说明、字段落地状态、上下游对象 | 不重复整张报表业务背景，不画复杂血缘图，不复制来源页证据全文 |
| 血缘页 | 技术视角说明：表级链路、字段级来源、加工逻辑、关系类型、缺口字段和影响分析 | 不承担监管业务定义主解释，不复制字段字典全文，不把无 SQL 来源的字段写成已闭环 |
| 概念页/系统页 | 沉淀跨报表复用规则、系统边界、通用术语和方法 | 不承载单张报表的明细字段清单 |

重复信息只保留“摘要 + 链接”。完整字段口径放数据表页，完整字段级来源放血缘页，证据包放来源页。

## 核心原则

1. 原始材料不可变，不要改写 `原始材料/` 下的文件。
2. 先表达业务含义，再描述 SQL 实现。
3. 任何重要结论都应能追溯到本地 SQL/DDL、外部 `my-dev-brain` wikilink或既有知识页。
4. 优先更新已有页面，不制造近似重复页。
5. 如果 SQL 语义有歧义，必须在相关页 `## Open Questions` 中显式写出。
6. 用户提出可复用的问题、澄清或结论时，应回写知识库，而不是只留在对话里。
7. 不复制大段监管原文，只记录外部 vault 名、Obsidian wikilink、检索关键词和提炼后的结论。
8. 所有 SQL、报表、指标、数据表、字段、来源、血缘页面都必须通过目录路径体现所属系统。
9. 系统间默认彼此独立；跨系统关系通过血缘页或映射说明表达，不通过共享路径表达。
10. 字段口径和指标口径统一写入数据表页，不再维护独立字段页或指标页。
11. 代码类字段的码值含义、取值范围和映射规则，默认直接写入接口表或结果表对应字段口径，不再单独维护码值页。
12. ingest、query、lint、SQL 开发都必须优先更新已有页面和共享知识页。
13. 新来源如果挑战旧结论，必须显式标注冲突、更新相关页面，并在必要时降级页面状态；不能静默覆盖。
14. 不允许在知识页中制造指向不存在页面的占位 wikilink；待补内容写成普通文本“待补充”或“待补（目标页名）”。
15. agent 生成的新 SQL 默认放入 `工作区/SQL开发/<系统名>/`。只有用户确认其已成为正式本地材料后，才可按系统归档到 `原始材料/SQL/<系统名>/`，并按 SQL ingest 工作流沉淀知识。

## 必备 Frontmatter

每个知识库页面都应包含：

```yaml
---
type: report|table|lineage|source|concept
id: 报表-...
status: draft|validated
updated: YYYY-MM-DD
tags:
  - regulatory
  - <type-tag>
---
```

来源页必须额外包含：

```yaml
external_vault: my-dev-brain
external_paths:
  - "[[03-实体/...]]"
search_keywords: []
```

`tags` 第二项应随页面类型变化：报表业务口径页使用 `report`，数据表页使用 `table`，血缘页使用 `lineage`，来源页使用 `source`，概念页使用 `concept`。

如果本次 ingest 只有本地 SQL、DDL 或字段字典材料，尚未定位到外部监管来源，来源页仍可创建，但 `external_paths` 应留空数组 `[]`，并在 `search_keywords` 或 `## Open Questions` 中记录待检索关键词和缺口。

## 页面状态规则

- `draft`：
  - 新建页面默认使用 `draft`
  - 仍有关键证据缺口
  - 必填章节未完成
  - 仍存在阻塞型 `Open Questions`
  - 仍存在未修复死链、明显占位内容或互链不完整
- `validated`：
  - 必填章节已完成
  - 核心结论已能回溯到本地 SQL、外部来源 wikilink 或既有知识页
  - 互链、索引与日志已同步
  - 不再存在阻塞型缺口
  - 即使仍保留 `Open Questions`，也只能是非阻塞问题

新建或大幅改写页面后，agent 必须在本次工作末尾重新判断状态。如果新来源推翻或削弱既有 `validated` 结论，相关页面必须降回 `draft`。

## 工具使用规则

- `工具/检索外部知识库.py`：用于在 `my-dev-brain` 中按检索顺序寻找候选监管来源。找到候选后，仍需人工判断哪些页面应以 Obsidian wikilink 写入 `external_paths`。
- `工具/血缘脚手架.py`：用于 SQL ingest 的第一轮结构化抽取。只能作为脚手架，不能替代业务解释和字段级核对。
- agent 应优先复用仓库工具和模板，而不是每次手工从零开始整理。

## 索引与日志总规则

- `索引.md` 是总索引，只保留系统摘要、项目模块说明、任务路由和更新检核清单。
- 具体报表、数据表、来源和血缘入口应维护在对应系统页，例如 `知识库/概念/概念-系统-<系统名>.md`。
- 新增系统时，先创建或更新系统页，再在 `索引.md` 中新增一行系统摘要。
- 新增具体报表、数据表、来源页或血缘页时，默认更新对应系统页，不在根索引平铺。
- 每次 ingest、SQL 开发、查询回写、lint 或规范调整后，应向 `日志.md` 追加记录，说明触达对象、状态判断和遗留问题。

详细规则见 `.agents/rules/索引与日志规则.md`。

## 写作风格

- 业务解释优先使用简洁中文。
- 英文或 `snake_case` 只用于技术标识。
- 优先使用要点和表格，不写长篇大段说明。
- Obsidian 互链使用 `[[报表-对公存款余额日报]]` 格式；不存在页面不要写成 wikilink。
- 代码片段只保留证明结论所必需的最短部分。
- 对外部来源，只记录 `my-dev-brain` 的 vault 名和 Obsidian wikilink，不复制正文。

详细页面写作规则见 `.agents/rules/页面写作规则.md`。
