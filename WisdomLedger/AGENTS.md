# WisdomLedger 智能体规范

这个仓库是一个由 LLM 持续维护的知识库，用来把监管报表 SQL 沉淀为：

1. 业务可读的自然语言口径
2. 可持续维护的血缘文档
3. 可复用的报表与数据表知识

整体运行方式遵循 Karpathy 提出的 `LLM Wiki` 三层模式：

- `原始材料/`：不可变的原始材料层。本仓库只存 SQL 和本地技术材料。
- `知识库/`：由 LLM 维护的知识层。报表业务口径页、数据表页、血缘页是主载体。
- `AGENTS.md`：约束 LLM 工作方式的规范文件，用来保证结构、流程和维护纪律。

监管原文、填报说明、制度文件和指标来源材料不保存在本仓库，而是关联外部 Obsidian 知识库：

- 外部 vault：`my-dev-brain`
- 任何外部来源依赖都应记录为 Obsidian wikilink，例如 `[[03-实体/一表通-1.1机构信息]]`，不要记录本机文件系统绝对路径。

## 仓库结构

- `原始材料/SQL/`：按系统分目录存放报表 SQL、中间加工 SQL、视图定义等本地技术材料。
- `原始材料/表结构/`：按系统分目录存放表结构、字段字典、接口字段说明等本地技术材料。
- `知识库/来源/`：按系统分目录；每个来源包一页，记录本地 SQL 与外部监管来源的关联。
- `知识库/报表业务口径/`：按系统分目录；每张监管报表一页。
- `知识库/数据表/`：按系统分目录；每张物理表或稳定逻辑数据集一页，默认同时承载字段和表内指标口径。
- `知识库/血缘/`：按系统分目录；端到端血缘页。
- `知识库/概念/`：可复用的概念、术语和方法页。
- 系统页统一放在 `知识库/概念/` 下，命名为 `概念-系统-<系统名>.md`。
- `模板/`：页面模板。用于复制结构，不保留占位内容。
- `工作区/SQL开发/`：agent 根据业务需求生成的 SQL、存储过程草案、验证脚本和说明。草案确认前不放入 `原始材料/`。
- `索引.md`：内容导向的总目录。
- `日志.md`：按时间追加的操作记录。

## 路径分层规则

- 系统是路径层级，不只是页面正文属性。
- 除 `知识库/概念/` 外，所有知识页和 SQL 文件默认都应放在对应系统目录下。
- 不同系统的对象，即使同名，也不能放在同一路径下混用。
- 跨系统内容不建“混合目录”，而是放在主系统目录下，并在正文中显式写明涉及的其他系统。

推荐路径形态：

- `原始材料/SQL/<系统名>/...`
- `原始材料/表结构/<系统名>/...`
- `知识库/来源/<系统名>/来源-...`
- `知识库/报表业务口径/<系统名>/报表-...`
- `知识库/数据表/<系统名>/数据表-...`
- `知识库/血缘/<系统名>/血缘-...`
- `工作区/SQL开发/<系统名>/...`

例如：

- `原始材料/SQL/核心系统/对公存款余额日报.sql`
- `知识库/数据表/核心系统/数据表-客户主表.md`
- `知识库/血缘/监管报送平台/血缘-对公存款余额日报.md`
- `工作区/SQL开发/监管报送平台/PROC_对公存款余额日报_草案.sql`


## 命名规则

- 来源页：`来源-...`
- 报表业务口径页：`报表-...`
- 数据表页：`数据表-...`
- 血缘页：`血缘-...`
- 概念页：`概念-...`

页面标题优先使用中文业务名称。技术标识放在 frontmatter 或专门章节中。

目录路径优先使用系统名进行分层，页面标题不强制带系统名；只有在同名对象确实需要区分时，才在页面标题中追加系统名。

## 正式知识对象

- 单个报表 ingest 不只产出“报表包”。
- 以下对象都属于正式知识对象，agent 在 ingest、query、lint 时都必须检查是否需要更新：
  - 来源页
  - 报表业务口径页
  - 数据表页
  - 血缘页
  - 系统页与通用概念页
- 如果新信息具有跨报表复用价值，优先更新共享知识对象，而不是只写进当前报表业务口径页。

## 知识对象边界

为避免报表业务口径页、数据表页、血缘页和来源页互相复制，所有 agent 必须按以下边界写作：

| 对象 | 主职责 | 应避免 |
| --- | --- | --- |
| 来源页 | 记录证据包：本地 SQL/DDL/表结构材料、外部 `my-dev-brain` wikilink、检索关键词、关键发现、版本冲突 | 不维护完整字段口径，不复制报表字段表，不展开字段级血缘 |
| 报表业务口径页 | 业务视角说明：报表报什么、报谁、报送范围、纳入排除规则、字段或指标是否属于监管报送要求 | 不作为字段技术来源的唯一承载页，不复制数据表页完整字段字典，不展开完整字段级血缘 |
| 数据表页 | 字段与表内指标主档：主键、字段含义、码值、取值说明、字段落地状态、上下游对象 | 不重复整张报表业务背景，不画复杂血缘图，不复制来源页证据全文 |
| 血缘页 | 技术视角说明：表级链路、字段级来源、加工逻辑、关系类型、缺口字段和影响分析 | 不承担监管业务定义主解释，不复制字段字典全文，不把无 SQL 来源的字段写成已闭环 |
| 概念页/系统页 | 沉淀跨报表复用规则、系统边界、通用术语和方法 | 不承载单张报表的明细字段清单 |

重复信息处理规则：

- 必要重复只保留“摘要 + 链接”，完整内容放在主职责页面。
- 字段口径完整表默认放在数据表页；报表业务口径页只保留面向业务的字段清单和报送要求摘要。
- 字段级来源完整表默认放在血缘页；数据表页只写字段当前取值/落地状态，必要时链接血缘页。
- 证据原文、SQL 片段、外部来源 wikilink 默认放在来源页；其他页面只引用来源页。
- 如果同一结论必须在多页出现，应在非主页面写明“详见 [[数据表-...]] / [[血缘-...]] / [[来源-...]]”，不要整段复制。

## 核心原则

1. 原始材料不可变，不要改写 `原始材料/` 下的文件。
2. 先表达业务含义，再描述 SQL 实现。
3. 任何重要结论都应能追溯到证据：
   - 本地 SQL 文件
   - `my-dev-brain` 中的外部监管来源
   - 已存在的知识库页面
4. 优先更新已有页面，不要制造近似重复页。
5. 如果 SQL 语义有歧义，必须在 `## Open Questions` 中显式写出。
6. 当用户提出了可复用的问题和结论，应把它们回写到知识库，而不是只留在对话里。
7. 不要把大段监管原文复制到本仓库；这里只记录外部 vault 内 Obsidian wikilink、检索关键词和提炼后的结论。
8. 所有 SQL、报表、指标、数据表、字段、来源、血缘页面都必须通过目录路径体现所属系统。
9. 系统间默认彼此独立；跨系统关系通过血缘页或映射页表达，不通过共享路径表达。
10. 字段口径和指标口径统一写入数据表页，不再维护独立字段页或指标页。
11. 代码类字段的码值含义、取值范围和映射规则，默认直接写入接口表或结果表对应字段口径，不再单独维护码值页。
12. 如果同一套码值被多个接口表复用，也优先分别写入各自字段口径；不要为“复用”而强行抽成独立页面。
13. ingest、query、lint 都必须优先更新已有页面和共享知识页，而不是不断新增孤立页面。
14. 新来源如果挑战旧结论，必须显式标注冲突、更新相关页面，并在必要时降级页面状态；不能静默覆盖。
15. 不允许在知识页中制造指向不存在页面的占位 wikilink；待补内容一律写成普通文本“待补充”或“待补（目标页名）”。
16. 如仓库中仍存在 `原始材料/码值/`、`知识库/码值/` 或 `知识库/码值映射/`，仅视为历史遗留或原始证据入口；新增知识不得写入独立码值页或码值映射页。
17. 当用户要求编写 SQL、存储过程、视图、加工脚本或调度脚本时，必须优先利用已维护的数据表页、血缘页、报表业务口径页和来源页反推实现；不能只凭字段名猜测业务口径。
18. agent 生成的新 SQL 默认放入 `工作区/SQL开发/<系统名>/`；只有用户确认其已成为正式本地材料后，才可按系统归档到 `原始材料/SQL/<系统名>/`，并按 SQL ingest 工作流沉淀知识。

## 必备 Frontmatter

每个知识库页面都应包含：

```yaml
---
type: report|table|lineage|source|concept
id: 报表-...
status: draft|validated
updated: YYYY-MM-DD
tags:
  - regulatory
  - <type-tag>
---
```

来源页必须额外包含：

```yaml
external_vault: my-dev-brain
external_paths:
  - "[[03-实体/...]]"
search_keywords: []
```

除非确实稳定且有用，不要随意增加额外字段。

`tags` 第二项应随页面类型变化：报表业务口径页使用 `report`，数据表页使用 `table`，血缘页使用 `lineage`，来源页使用 `source`，概念页使用 `concept`。

来源页中的 `external_paths` 必须使用 Obsidian wikilink 形式，例如 `[[03-实体/一表通-1.1机构信息]]` 或 `[[01-资料库/一表通系统/2026-04-27-1.1机构信息-填报说明-原文]]`。外部来源依赖不要写本机文件系统绝对路径；如需定位外部库，只记录 vault 名 `my-dev-brain` 与库内 wikilink。

如果本次 ingest 只有本地 SQL、DDL 或字段字典材料，尚未定位到外部监管来源，来源页仍可创建，但 `external_paths` 应留空数组 `[]`，并在 `search_keywords` 或 `## Open Questions` 中记录待检索关键词和缺口；不要为了填充 `external_paths` 制造不存在或未核验的 wikilink。

## 页面状态规则

- `draft`：
  - 新建页面默认使用 `draft`
  - 仍有关键证据缺口
  - 必填章节未完成
  - 仍存在阻塞型 `Open Questions`
  - 仍存在未修复死链、明显占位内容或互链不完整
- `validated`：
  - 必填章节已完成
  - 核心结论已能回溯到本地 SQL、外部来源 wikilink 或既有知识页
  - 互链、索引与日志已同步
  - 不再存在阻塞型缺口
  - 即使仍保留 `Open Questions`，也只能是非阻塞问题
- 状态变更规则：
  - 新建或大幅改写页面后，agent 必须在本次工作末尾重新判断状态
  - 如果新来源推翻或削弱既有 `validated` 结论，相关页面必须降回 `draft`
  - 不能把 `validated` 当作“写完了”的装饰标签；它代表当前版本已经完成最小验证闭环

## 工具使用规则

- `工具/检索外部知识库.py`：
  - 用于在 `my-dev-brain` 中按检索顺序寻找候选监管来源
  - 找到候选后，仍需人工判断哪些页面应以 Obsidian wikilink 写入 `external_paths`
- `工具/血缘脚手架.py`：
  - 用于 SQL ingest 的第一轮结构化抽取
  - 只能作为脚手架，不能替代业务解释和字段级核对
- agent 应优先复用这些工具，而不是每次都手工从零开始整理。

## Ingest 触发边界

当用户说“ingest”但没有明确对象时，agent 必须先根据输入材料判断触发哪条流程：

| 输入材料或用户意图                                | 默认流程              | 主要产出                          | 不应默认产出                |
| ---------------------------------------- | ----------------- | ----------------------------- | --------------------- |
| DDL、字段字典、接口字段说明、表结构截图或“先初始化数据表”          | 数据表 Ingest 工作流    | 来源页、数据表页、索引、日志，必要时更新系统页/概念页   | 不硬造报表业务口径页，不硬造完整血缘        |
| 报表 SQL、存储过程、视图定义、结果表加工脚本或“ingest 报表 SQL” | 报表 SQL Ingest 工作流 | 来源页、报表业务口径页、数据表页、血缘页、索引、日志        | 不跳过已有数据表页，不只生成 SQL 摘要 |
| 用户要求“写 SQL”“写存储过程”“按这些表实现需求”“生成加工脚本”“补 SQL” | SQL/存储过程开发工作流 | `工作区/SQL开发/<系统名>/` 下的 SQL 草案、验证 SQL、实现说明，并同步维护报表业务口径页、数据表页、血缘页、来源页、索引和日志；用户明确要求时可同步更新 `原始材料/SQL/<系统名>/` 既有 SQL | 默认不直接改写 `原始材料/`，不把未验证草案写成已验证生产血缘 |
| 用户只问口径、差异、字段含义或历史结论                      | 查询工作流；若结论可复用再回写   | 回答、必要的知识页更新、索引、日志             | 不默认新建整套报表包            |
| 用户要求巡检、检查、lint、找死链或看一致性                  | 巡检工作流             | 修复项、Open Questions、状态调整、索引、日志 | 不把 lint 当作只读清单        |

如果同一批材料同时包含表结构和 SQL，应先执行数据表初始化或更新，再执行报表 SQL ingest，并在日志中写明两阶段都已处理。

## 数据表 Ingest 工作流

当用户要求 ingest 数据表、表结构、字段字典、接口字段说明或先初始化数据表知识时，按以下流程处理。该流程可以先于报表 SQL ingest 独立执行，用来建立数据表主档；后续 SQL ingest 再补充加工逻辑、字段级血缘和报表使用关系。

1. 读取 `原始材料/表结构/<系统名>/` 下的 DDL、字段字典、接口说明或其他本地表结构材料，不改写原始材料。
2. 先在本库搜索同系统下已有数据表页、来源页、血缘页、报表业务口径页、系统页和概念页，优先更新已有对象，不要默认新建近似重复表页。
3. 判断数据表所属系统、表类型和知识边界：
   - 物理表
   - 稳定逻辑数据集
   - 接口表
   - 结果表
   - 监管报送表
4. 创建或更新 `知识库/来源/<系统名>/来源-...`，记录本地 DDL、字段字典、接口说明、已核验的外部 vault `my-dev-brain` Obsidian wikilink（如有）、检索关键词和关键发现；不复制大段原文，不写外部绝对路径。
5. 创建或更新 `知识库/数据表/<系统名>/数据表-...`，至少写清：
   - 表在监管链路中的角色
   - 主键与核心标识
   - 字段清单与字段口径
   - 字段落地状态
   - 表内指标口径（如有）
   - 上游对象、下游对象和相关报表
   - 相关来源页和血缘页
6. 字段落地状态必须区分：
   - 本地表结构字段：DDL 或字段字典中已经存在
   - SQL 已加工字段：后续 SQL、过程或视图中已经明确赋值、映射或聚合
   - 当前缺口字段：表结构已存在，但 SQL 来源、填报规则或是否报送尚未闭环
   - 不适用字段：有明确依据说明当前场景不需要报送或不参与加工
   - 监管要求字段：仅当外部监管来源或既有报表业务口径页已经确认需要报送时才标注
7. 如果该表是接口表、结果表或监管报送表，所有代码类字段必须在字段口径中直接写清码值、中文含义、映射规则、特殊值或常量值、证据来源；不得只写“见码表”或新建独立码值页。
8. 如果当前只有表结构材料、尚无 SQL 加工证据，不要硬造血缘；数据表页中标为“待补血缘”，必要时在 `## Open Questions` 中记录缺口。
9. 如果已有相关血缘页，补齐数据表页到血缘页的回链；如果尚无血缘页，只有在已有明确上下游证据时才创建或更新血缘页，否则写成普通文本“待补（血缘页名）”，不要制造不存在 wikilink。
10. 检查是否产生可复用知识，例如系统边界、通用表角色、共享字段规则；如有，更新系统页或通用概念页。
11. 对照既有页面检查是否出现字段定义冲突、码值冲突、表归属变化或旧结论被削弱；冲突可确认时直接更新旧页并写明影响，不能确认时写入相关页 `## Open Questions`。
12. 重新判断触达页面的 `status`。仅完成表结构初始化、但缺该表应有的 SQL 加工、上下游或关键监管证据时，数据表页通常保持 `draft`；如果该表只是上游基础表，且字段字典、主键、角色、来源和已知上下游已经闭环，可以评估为 `validated`。
13. 更新 `索引.md`。
14. 向 `日志.md` 追加一条时间记录，记录本次数据表 ingest 触达了哪些表、来源、血缘或共享知识页，以及是否存在缺口或冲突。

说明：

- 数据表 ingest 的目标是先建立“字段口径主档”，不要求同时完成报表业务口径页和完整血缘页。
- 后续报表 SQL ingest 时，必须回到已初始化的数据表页补充 SQL 加工状态、上下游关系、字段级血缘和相关报表链接。
- 如果用户只提供表结构，不要推断监管报送范围；是否需要报送应以后续外部监管来源、报表业务口径页或 SQL 证据为准。
- 数据表 ingest 完成的最低标准是：一张来源页、一张数据表页、字段清单和字段落地状态、必要的 Open Questions、索引更新、日志记录；如触达接口表、结果表或监管报送表，还必须覆盖代码类字段码值说明。

## 报表 SQL Ingest 工作流

当用户要求 ingest 一份新的监管报表 SQL 时：

1. 读取 `原始材料/SQL/<系统名>/` 下的新 SQL 文件；必要时先用 `工具/血缘脚手架.py` 做第一轮结构抽取。
2. 在本库中先搜索相关报表业务口径页、数据表页、血缘页、系统页和概念页，优先更新已有对象，不要默认新建。
3. 用 `工具/检索外部知识库.py` 或按既定顺序检索外部 vault `my-dev-brain`，定位报表业务口径页、主题页和原文页；写入知识页时只记录 Obsidian wikilink。
4. 创建或更新一页 `知识库/来源/<系统名>/来源-...`，记录本地文件、外部来源 wikilink（如有）、检索关键词和关键结论，不复制原文。
5. 创建或更新对应的 `知识库/报表业务口径/<系统名>/报表-...` 页面。
6. 为涉及的核心物理表和稳定逻辑数据集创建或更新 `知识库/数据表/<系统名>/` 下的数据表页，并把字段口径、指标口径写回这些表页。
7. 如果目标对象是接口表、结果表或监管报送表，所有代码类字段都必须在对应字段口径中直接写明：
   - 码值
   - 中文含义
   - 映射规则
   - 特殊值或常量值
   - 证据来源
8. 创建或更新 `知识库/血缘/<系统名>/血缘-...` 页面，至少覆盖目标输出字段的字段级来源。
9. 检查这次 ingest 是否产生了跨报表复用知识：
   - 新的系统边界说明
   - 新的通用概念
   如果有，必须同步更新对应共享知识页，而不是只留在当前报表包里。
10. 对照既有知识页检查是否出现口径冲突、旧结论被替代、旧链接失效或系统边界变化：
   - 如果冲突可确认，直接更新旧页并写明变更影响
   - 如果冲突尚不能确认，在相关页 `Open Questions` 中显式记录
   - 如果既有 `validated` 页面被新来源削弱，降为 `draft`
11. 补齐报表业务口径页、数据表页、血缘页、来源页、系统页、概念页之间的双向互链。
12. 重新判断这次触达页面的 `status`。
13. 更新 `索引.md`。
14. 向 `日志.md` 追加一条时间记录，记录本次 ingest 触达了哪些关键对象，以及是否发现冲突或补了共享知识页。

说明：

- 一次 ingest 可以也应该同时更新多页；不要把它理解成“一个 SQL 只配一个报表包”。
- 如果一份新来源只生成了来源页、报表业务口径页、数据表页和血缘页，却没有检查共享概念、系统页、接口表字段码值说明和旧结论冲突，默认视为 ingest 未完成。

## SQL/存储过程开发工作流

当用户要求写 SQL、生成 SQL、补 SQL、写存储过程、生成视图、生成加工脚本或校验 SQL 时，应优先使用 `.agents/skills/sql-dev/` 中的 SQL 开发 skill；同时必须遵守本 `AGENTS.md` 中关于路径分层、知识页维护、索引、日志、状态和原始材料不可变的规则。

`AGENTS.md` 负责仓库级约束；`.agents/skills/sql-dev/` 负责 SQL 开发执行流程、SQL 风格和交付检查清单。如两者冲突，以本 `AGENTS.md` 的仓库级规则为准。

当用户要求根据已维护的数据表完成业务需求、编写 SQL、编写存储过程、补加工逻辑、生成视图或调度脚本时，按以下流程处理。

1. 先澄清或自行抽取需求要素：
   - 目标产物：查询 SQL、`insert select`、存储过程、视图、临时表加工、校验 SQL 或调度片段。
   - 目标系统与目标表：必须落在明确的系统目录下；跨系统时选择主系统目录并说明涉及系统。
   - 业务目标：统计对象、数据粒度、时间口径、纳入规则、排除规则、输出字段、刷新方式。
   - 运行约束：数据库方言、跑批日期参数、是否先删后插、是否幂等、是否需要增量处理、性能限制。
2. 先读 `索引.md`，再打开相关数据表页、血缘页、报表业务口径页、来源页和系统页；如果知识页缺失或字段口径不完整，优先读取 `原始材料/表结构/<系统名>/` 下的 DDL 或字段字典补充依据。
3. 如果需求依赖监管报送口径、字段是否必报、码值含义或制度解释，应按外部来源检索顺序检索 `my-dev-brain`，并只记录 Obsidian wikilink；不要复制监管原文。
4. 先输出或记录“实现口径确认”：
   - 目标对象与数据粒度
   - 使用的数据表及其角色
   - 主连接键与时间条件
   - 关键过滤条件
   - 字段映射、常量、码值转换和派生字段
   - 无法确认的假设和风险
5. 生成 SQL 前必须检查目标字段是否均可追溯：
   - 已在数据表页或 DDL 中存在的字段，可以用于 SQL。
   - 仅在业务描述中出现、但知识库和表结构中没有证据的字段，不得硬写；应放入 `Open Questions` 或实现说明的“待确认字段”。
   - 码值字段必须使用数据表页、报表业务口径页或来源页中的既有码值说明；缺失时写成待确认，不得编造枚举。
6. SQL 草案默认写入 `工作区/SQL开发/<系统名>/`。如果用户明确要求修改 `原始材料/SQL/<系统名>/` 中的既有 SQL，可以按用户要求修改原始 SQL，但必须在来源页和日志中记录这次人工确认的改写，不得静默覆盖历史语义。推荐命名：
   - 查询脚本：`SQL_<需求名>_草案.sql`
   - 存储过程：`PROC_<需求名>_草案.sql`
   - 视图定义：`VIEW_<需求名>_草案.sql`
   - 校验脚本：`CHECK_<需求名>_校验.sql`
   - 实现说明：`说明-<需求名>.md`
7. SQL 文件必须尽量使用 `模板/SQL开发模板.sql` 的结构，至少包含：
   - 文件头：业务目标、依赖知识页、源表、目标表、参数、运行方式、未确认点。
   - 参数区：跑批日期、机构范围、重跑范围等。
   - 清理区：如需要先删后插，删除条件必须与插入范围一致。
   - 主加工区：默认使用直接 `select` + `left join` 组织；只有在多阶段汇总、窗口去重、复杂复用或可读性明显改善时才使用 CTE。
   - 落地区：目标字段顺序必须与目标表或输出清单一致。
   - 校验区：行数、主键重复、必填为空、码值越界、金额或余额方向等检查。
8. 存储过程必须额外满足：
   - 入参、出参、异常处理、提交/回滚策略明确。
   - 删除重跑、增量追加、截面刷新三种方式必须写明选择理由。
   - 过程名、临时表名、中间表名应带业务含义，避免只用 `tmp1/tmp2`。
   - 对大表关联要明确驱动表、过滤前置、必要索引字段和避免全表笛卡尔积的措施。
9. 生成 SQL 后必须同时给出验证 SQL 或验证清单，至少覆盖：
   - 目标行数是否符合业务范围。
   - 目标主键或唯一性是否重复。
   - 必填字段是否为空。
   - 代码类字段是否只出现允许码值。
   - 金额、余额、日期字段是否存在明显异常。
   - 与上游来源的抽样回溯是否能对上。
10. 开发完成后必须同步维护相关知识页，不能只交付 SQL：
   - 来源页：创建或更新 `知识库/来源/<系统名>/来源-...`，记录本次开发草案路径、依赖的表结构/知识页、外部来源 wikilink（如有）、关键实现结论和待验证点。
   - 报表业务口径页：如果 SQL 对应报表、接口、结果表或固定业务输出，必须创建或更新 `知识库/报表业务口径/<系统名>/报表-...`，写清整表口径、字段/指标口径、纳入排除规则、时间口径和代码类字段规则。
   - 数据表页：更新所有被读取或写入的核心数据表页，补充字段使用关系、SQL 已加工字段、下游对象、相关报表和待确认字段。
   - 血缘页：创建或更新 `知识库/血缘/<系统名>/血缘-...`，至少覆盖本次 SQL 的表级链路和目标输出字段的字段级来源。
   - 概念页/系统页：如果产生新的可复用加工规则、系统边界、通用码值解释或日期口径，应更新共享知识页。
   - `索引.md`：补齐新增或更新的来源、报表、数据表、血缘和概念入口。
   - `日志.md`：追加 `sql-dev` 或 `development` 记录，说明本次生成的 SQL、维护的知识页、验证状态和遗留问题。
11. 草案阶段与正式阶段必须区分：
   - SQL 尚未运行或尚未被用户确认时，相关知识页可以记录“SQL 草案待验证”，但页面状态通常保持 `draft`。
   - 血缘页可以写字段级设计血缘，但必须在证据或状态中标注“依据 SQL 草案”，不要写成已验证生产血缘。
   - 用户确认 SQL 已运行通过、已投产或已归档为正式材料后，必须按报表 SQL ingest 工作流重新判断状态，补齐来源页、报表业务口径页、数据表页、血缘页、索引和日志。
12. 如果开发过程中发现既有数据表页字段含义、码值、上下游或 Open Questions 已被用户澄清，应回写到相应知识页，并更新 `updated`、`status`、`索引.md` 和 `日志.md`。
13. 交付时必须说明：
   - 生成了哪些 SQL/说明文件。
   - 同步维护了哪些来源页、报表业务口径页、数据表页、血缘页或概念页。
   - 依据了哪些知识页或本地表结构材料。
   - 哪些假设仍需用户确认。
   - 建议先执行哪些校验 SQL。

### SQL 编写规则

- 业务口径优先于技术写法；SQL 注释必须说明业务含义，而不是机械复述语法。
- 默认不使用 `select *`；查询、落地、校验都应只查询实际使用的字段。
- 默认不使用多层 `with` / CTE；普通取数、补字段、码值映射和常规关联优先写成主表 `left join`。
- 使用 `with` / CTE 必须有明确理由，例如同一中间结果被多次复用、需要先聚合再关联、需要窗口函数去重、或直接 `left join` 会明显降低可读性。
- 目标字段顺序必须与目标表 DDL、数据表页字段清单或用户指定输出顺序一致。
- 所有常量字段、码值映射、条件判断、拼接字段、窗口函数和聚合字段都必须有注释说明依据。
- 金额、余额、日期、机构、客户、账号、合同号等关键字段必须明确来源表和来源字段。
- 关联条件必须写清业务键，不能只凭同名字段关联。
- 过滤条件必须能映射到业务纳入/排除规则；如果只是技术限制，要在注释中说明。
- 对日期口径要区分交易日、自然日、数据日期、会计日期、跑批日期和报送期。
- 对删除重跑脚本，`delete` 范围必须与 `insert` 范围一致，并优先使用跑批日期、报送期、机构等分区条件。
- 对汇总 SQL，必须说明分组粒度、去重逻辑、分子分母、聚合函数和空值处理。
- 对明细 SQL，必须说明每条记录代表什么业务对象，以及如何保证一业务对象一行或多行的规则。
- 数据库方言不明确时，优先写 ANSI SQL 风格，并在文件头标注“数据库方言待确认”；不要混用不同数据库专属语法。

### SQL 开发最低交付标准

一次 SQL/存储过程开发完成的最低标准是：

- 一份位于 `工作区/SQL开发/<系统名>/` 的 SQL 或存储过程草案。
- 必要时一份同目录校验 SQL。
- 文件头列明业务目标、依赖知识页、源表、目标表、参数、运行方式和未确认点。
- 关键字段映射、过滤条件、码值转换和聚合逻辑均有依据说明。
- 已同步维护相关来源页、报表业务口径页、数据表页和血缘页；如果确无对应对象，必须在交付说明中写明原因。
- 已更新必要的 `索引.md` 和 `日志.md`。
- 回答中列出依据页面和待确认问题。
- 如果产生长期可复用的新口径，已回写相关概念页或系统页。

## 如何把 SQL 翻译成业务语言

对于每一份 SQL，先抽取：

- 报表目的
- 统计粒度
- 维度
- 度量
- 纳入规则
- 排除规则
- 时间逻辑
- 来源数据表
- 关联逻辑
- 转换阶段
- 最终输出字段

然后把它改写成业务语言，重点回答：

- “到底在统计什么？”
- “统计发生在什么日期或粒度上？”
- “哪些对象被纳入，哪些被排除？”
- “哪些来源表是权威来源？”
- “哪些过滤条件会实质性改变业务口径？”

不要把 SQL 直接原样搬进正文。先用业务语言解释含义，再把 SQL 表达式作为证据保留。

对于监管原文：

- 不要在这里复制全文
- 只保留简洁的业务结论
- 记录 `my-dev-brain` 中的精确 Obsidian wikilink
- 记录可重复使用的检索关键词

## 报表业务口径页写法规则

报表业务口径页默认面向业务人员阅读，不要把报表业务口径页写成 SQL 注释翻译稿或开发说明。

### 明细类报表

如果目标报表是明细类报表，必须完整覆盖两部分：

1. 整表报送范围
2. 每个报送字段的字段级口径

最低要求：

- 必须说明整张表到底报送哪些业务对象
- 必须说明哪些对象纳入、哪些对象排除
- 必须说明这是全量报送、增量报送还是截面报送
- 必须逐字段说明口径，不能只写重点字段
- 每个字段都要说明：
  - 字段含义
  - 取值或映射逻辑
  - 来源说明
  - 特殊规则

明细类字段口径中的“来源说明”应优先使用自然中文句式，例如：

- `来源于监管集市机构表的机构地址字段`
- `来源于客户主表的客户编号字段`

不要在该列直接裸写技术表名和字段名，如：

- 不推荐：`ORG_ADD / L_PUBL_ORG_BRA`

如果确实需要保留技术标识，应放在“规则说明”中作为证据补充，而不是替代业务表达。

### 汇总类报表

如果目标报表是汇总类报表，必须完整覆盖每个指标的取数规则，不能只写核心指标。

最低要求：

- 必须逐指标说明，不可省略
- 每个指标都要说明：
  - 指标名称
  - 统计对象
  - 取数规则
  - 过滤条件
  - 聚合方式
  - 依赖来源
- 如果是派生指标，必须说明依赖哪些基础指标
- 如果存在分子、分母、去重口径、时点口径，必须写清楚

### 通用要求

- 业务解释优先，SQL 细节后置
- 不允许报表业务口径页只停留在“源表 + where 条件 + insert 目标表”的摘要层
- 代码类字段必须写清楚码值或映射来源
- 常量字段、拼接字段、判断字段、窗口函数字段都必须显式说明
- 如果页面中出现技术字段名，应同时给出业务名称或中文含义
- 模板以 `模板/报表业务口径模板.md` 为准，新增报表业务口径页默认按该模板落地

### 一表通字段版本判断

一表通报表常出现“监管填报说明已要求、目标表结构已建、但当前 SQL 尚未加工”的版本差异。agent 不能只凭 SQL 中没有字段就判断“无需报送”，也不能只凭 DDL 中有字段就判断“已经落地”。

遇到一表通字段增删或用户补充 DDL 时，必须区分四类状态：

- 监管要求字段：外部 `my-dev-brain` 填报说明或实体页确认需要报送的字段。
- 本地表结构字段：本仓库 DDL 或表结构材料中已经存在的字段。
- SQL 已加工字段：本仓库 SQL、过程或视图中已经明确赋值、映射或聚合的字段。
- 当前缺口字段：监管要求或表结构已存在，但 SQL 来源、填报规则或是否报送尚未闭环的字段。

写页规则：

- 报表业务口径页说明“是否需要报送”时，以外部监管来源和实体页为优先依据。
- 数据表页说明“是否已落地”时，同时列出表结构状态和 SQL 加工状态。
- 血缘页只把 SQL 可追溯字段标为已闭环；对尚无 SQL 来源的字段写成“待补血缘”，不要硬造来源。
- 如果用户明确说“某字段本行不需要报”，也必须记录依据、适用范围和是否影响监管要求；没有证据时写入 `Open Questions`。

## 数据表页写法规则

数据表页是字段口径和表内指标口径的默认承载页。

最低要求：

- 必须说明该表在监管链路中的角色
- 必须覆盖：
  - 主键与核心标识
  - 关键字段与口径
  - 表内指标口径（如有）
  - 上下游对象
- 字段和表内指标统一在数据表页内维护，不再拆成独立页面

如果该表是接口表、结果表或监管报送表，最低要求再增加：

- 代码类字段必须逐字段写清码值含义
- 不能只写“见码表”或“参考字典”
- 必须把常量值、枚举值、压缩映射和特殊取值直接写到字段口径中
- 如果某个字段的码值很多，可在该字段下补充子表，但仍需留在本表页内

## 血缘规则

每张血缘页都应包含：

1. 一段业务链路摘要
2. 节点列表
3. 表级边列表
4. 字段级边列表
5. 条件允许时提供 Mermaid 图
6. 已知缺口、人工假设或未确认点

血缘页默认面向技术人员使用，主要服务于：

- 影响分析
- 反向溯源
- 字段口径核对
- 变更评估

因此血缘页不能只停留在“表到表”或“CTE 到 CTE”的粗粒度摘要层。

并区分三类血缘：

- 物理血缘：表到表
- 逻辑血缘：CTE、中间结果、派生指标
- 语义血缘：监管指标 -> 业务指标 -> SQL 表达式 -> 来源字段

### 表级血缘要求

- 说明源表、目标表、中间层、过程或脚本之间的关系
- 说明每条边对应的主要处理动作，例如过滤、关联、聚合、插入、删除重算
- 表级图用于快速看全链路，不替代字段级血缘

### 字段级血缘要求

如果血缘页对应监管报表、标准接口表、核心下游结果表，默认应补充字段级血缘。

最低要求：

- 至少覆盖所有目标输出字段
- 每个目标字段都要说明来源字段、来源对象、处理逻辑
- 如果是直接映射，要明确写“直接映射”
- 如果是拼接、条件判断、码值映射、聚合、窗口函数、常量赋值，也要显式写出
- 如果多个目标字段来自同一源字段，不能省略任何一个目标字段
- 如果一个目标字段依赖多个源字段，要全部列出

字段级边列表建议至少包含：

- 源对象
- 源字段
- 目标对象
- 目标字段
- 处理逻辑
- 关系类型
- 证据

其中关系类型建议使用：

- 直接映射
- 条件映射
- 拼接派生
- 聚合派生
- 窗口派生
- 常量赋值
- 码值转换

字段级血缘默认面向技术人员，因此字段级边列表和字段级 Graph 中：

- 源字段、目标字段优先使用英文字段名或技术字段名
- 不要把中文字段名作为字段级血缘的主展示名称
- 如有必要，可在“处理逻辑”或补充说明中附带中文含义
- 推荐写法：
  - 源对象：中文或稳定技术对象名均可
  - 源字段：`ORG_ADD`
  - 目标字段：`A010016`

### Graph 画法要求

- 每张血缘页建议至少提供两层图：
  - 总览图：展示表级和过程级主链路
  - 详细图：展示字段级关联关系
- 字段级图应尽量体现“原表.原字段 -> 目标表.目标字段”
- 字段级图必须逐字段拆分：一个 Mermaid 节点只放一个源字段或一个目标字段；不要把多个字段合并成 `A/B/C -> X/Y/Z` 这种聚合节点。
- 如果一个目标字段依赖多个源字段，应画多条边指向同一个目标字段；如果一个源字段派生多个目标字段，应画多条边从同一个源字段分别指向各目标字段。
- 如果字段过多，可以分主题拆成多张 Mermaid 图，但不能省略目标字段
- 如果同一字段存在多来源回溯，应在图或边表中写清“本级/上级/回溯补齐”等关系
- 图是为技术分析服务，优先完整和可追溯，不必追求简洁美观

### 血缘互链规则

为了支持影响分析和反向溯源，血缘关系不能只存在于血缘页本身，必须形成对象页与血缘页的双向回链。

最低要求：

- 每张血缘页都要显式列出：
  - 直接上游对象
  - 直接下游对象
  - 相关数据表页
  - 相关报表业务口径页
- 每个被血缘页引用的核心数据表页，都应回链到对应血缘页
- 每个被血缘页引用的核心报表业务口径页，都应回链到对应血缘页
推荐理解方式：

- 从血缘页出发，能跳到上游对象和下游对象
- 从数据表页出发，能看到“我被哪些血缘页使用”
- 从报表业务口径页出发，能看到“这张报表对应哪条血缘链路”
### 模板联动要求

在创建或更新以下页面时，应同步检查互链是否完整：

- `模板/血缘模板.md`
- `模板/数据表模板.md`
- `模板/报表业务口径模板.md`
- `模板/来源模板.md`

如果新增了一张血缘页，至少要同步检查：

1. 上游数据表页是否已回链该血缘页
2. 下游数据表页是否已回链该血缘页
3. 报表业务口径页是否已回链该血缘页
4. 上下游表页和报表业务口径页是否已补齐回链

## 查询工作流

当回答问题时：

1. 先读 `索引.md`
2. 打开最相关的知识库页面
3. 如果问题依赖监管原文含义，再去检索 `my-dev-brain`
4. 结合页面和外部来源做综合回答
5. 回答中必须显式给出依据
6. 如果答案具有长期价值，把它回写到知识库

### Query 引用规则

- 回答中的关键结论不能裸写，至少要带一条本库引用。
- 本库引用优先使用页面名，例如：
  - `依据：[[报表-...]]`
  - `依据：[[数据表-...]]、[[血缘-...]]`
- 如果结论直接依赖外部监管原文或 `my-dev-brain` 的页面含义，必须同时给出 `external_paths` 中的 Obsidian wikilink 命中来源。
- 如果回答综合了多个页面，应在回答结尾集中列出“引用页面”或“依据页面”。
- 如果当前知识库里还没有合适承载页，但这个问答具有长期复用价值，应创建或更新下列之一：
  - 相关报表业务口径页
  - 相关数据表页
  - 相关血缘页
  - 通用概念页
  - 比较页

### Query 回写规则

- 满足以下任一条件，就不应只停留在对话里：
  - 用户问到了可复用的口径差异
  - 用户澄清了一个长期存在的歧义
  - 回答形成了跨报表可复用的结论
  - 回答发现了既有页面的错误、过时或矛盾
- 回写后要同步更新相关页面的 `updated`、`status`、`索引.md` 和必要的 `日志.md`。

## 巡检工作流

lint 不是泛泛“看一眼”，而是一种可执行维护动作。默认输入为：

- `索引.md`
- `日志.md`
- 最近修改页
- 相关来源页、报表业务口径页、数据表页、血缘页、概念页

默认检查项包括：

- 孤儿页面或几乎没有入链的页面
- 缺少上下游链接的数据表页
- 在血缘页中提到、但还没有沉淀到表页或概念页的关键对象或规则
- 业务定义与 SQL 过滤条件是否矛盾
- 是否存在被新 SQL 或新规则替代的陈旧页面
- 来源页是否缺少 `external_paths` 字段，或在未定位外部来源时没有记录待检索缺口
- 是否存在指向不存在页面的 wikilink
- 是否仍有“待补充”出现在必填章节
- `validated` 页面是否仍满足验证条件
- 接口表中的代码类字段是否已写清码值说明

### Lint 处理规则

- 对于可直接修复的问题，agent 应直接修复，而不是只列问题清单。
- 对于暂时无法修复的问题，必须：
  - 写入相关页面的 `Open Questions`
  - 在必要时降级页面状态
  - 在 `日志.md` 中记录一次 `audit` 或 `lint` 条目
- Lint 如果修了互链、状态、索引或共享知识页，也要同步更新这些对象。
- 死链规则：
  - 知识页中的不存在 wikilink 视为错误
  - `日志.md`、`AGENTS.md`、`模板/` 中作为历史记录或示例出现的占位链接，不纳入知识图完整性判断
  - 如果对象尚未建立，使用普通文本 `待补充` 或 `待补（目标页名）`，不要制造空链接

### Lint 输出要求

- 如果本次 lint 没有改文件，也要给出明确结论：检查范围、未发现问题或仍存风险
- 如果本次 lint 改了文件，至少要更新：
  - 被修复页面
  - 必要的 `索引.md`
  - 必要的 `日志.md`
  - 被影响页面的 `status`

## 写作风格

- 业务解释优先使用简洁中文
- 英文或 `snake_case` 只用于技术标识
- 优先使用要点和表格，不写长篇大段说明
- Obsidian 互链使用 `[[报表-对公存款余额日报]]` 这种格式
- 代码片段只保留证明结论所必需的最短部分
- 对外部来源，只记录 `my-dev-brain` 的 vault 名和 Obsidian wikilink，不复制正文

## 每张新报表的最低交付标准

一份报表 ingest 完成的最低标准是：

- 一张带 `external_paths` 字段的来源页；如未定位外部来源，`external_paths` 可为空数组，但必须记录待检索缺口
- 一张报表业务口径页
- 一张血缘页
- 所有核心数据表页
- 核心字段和核心指标口径已经写入数据表页
- 接口表中的代码类字段已写入字段口径，不依赖独立码值页
- 相关共享知识页已经检查，并在需要时更新：
  - 系统页
  - 通用概念页
- 相关页面的 `status` 已重新判断
- 已更新的 `索引.md` 和 `日志.md`

独立指标页或字段页不再维护。
